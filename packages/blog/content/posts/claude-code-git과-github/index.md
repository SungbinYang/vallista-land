## 🐙 Git, Github 기초부터 실전까지!

[지난 섹션](/claude-code-슬래시-명령어와-단축키)까지 클로드 코드를 *내 손에 맞게* 길들였다. 이제 개발에서 *빼놓을 수 없는* 도구, **Git과 GitHub** 차례다.
[설치 챕터](/claude-code-개발환경-및-클로드-코드-설치)에서 *"소스 코드의 타임머신"* 으로 맛만 봤다면, 이번엔 **실전 사용법** 을 제대로 익힌다. 강의는 Cursor로 진행하지만, 우리는
*[손에 익은](/claude-code-cursor-ai-ide-통합)* **WebStorm** 으로 따라간다.

> Git이 익숙하지 않으신 분들은 *터미널 명령어나 모르는 용어가 나와도* **너무 걱정하실 필요 없어요.** … 정말 *자연스러워집니다.*

### Git은 왜, GitHub는 왜

먼저 *두 도구의 역할* 부터 짚자.

> **Git** 이 바로 *소스코드의 변경사항을 기록하고 관리* 해주는 도구라고 보시면 됩니다.

개발을 하다 보면 *로그인 기능 → 버그 수정 → 관리자 기능* 처럼 끝없이 *변경* 이 쌓인다. Git은 이 변경 *이력* 을 기록해, **언제든 원하는 시점으로** 되돌아가게 해준다. 그런데 한 가지 *불안* 이
있다.

> 로컬 PC에 저장되어 있기 때문에, *노트북을 잃어버린다거나* 실수로 기록을 제거하게 되면 **정말 불안하겠죠?**

그래서 **GitHub** 가 필요하다. *Git으로 관리되는 프로젝트를 온라인에 저장* 하는 공간이다. **안전한 백업** 이자, 남들과 *공유·협업* 하는 무대이기도 하다.

| 도구       | 역할                                        |
|------------|---------------------------------------------|
| **Git**    | *내 PC* 에서 변경 이력을 기록·관리 (로컬)   |
| **GitHub** | Git 프로젝트를 *온라인* 에 보관·공유 (원격) |

### 1. Git 시작하기 — `git init`

Git으로 프로젝트를 관리하려면, 먼저 그 폴더를 **Git 저장소로 초기화** 해야 한다.

```bash
git init    # 현재 폴더를 Git 저장소로 초기화
```

이러면 폴더 안에 **`.git`** 이라는 *숨김 디렉터리* 가 생긴다. *변경 이력* 이 전부 여기 담긴다. (이 폴더를 지우면 *이력도 사라지니* 건드리지 말자.)

**WebStorm** 에서는 *터미널 없이* GUI로도 된다.

- 상단 메뉴 **`VCS` → `Enable Version Control Integration…`** → *Git* 선택
- 또는 이미 Git 메뉴가 있다면 **`Git → Init…`**

초기화하면 WebStorm 좌측 파일들이 *색깔로* 표시되며 **버전 관리가 시작** 된다.

### 2. 커밋 (Commit) — "이 순간을 저장"

Git의 핵심은 **커밋 (Commit)** 이다. *프로젝트의 특정 시점 상태* 를 **기록** 하는 것, 게임으로 치면 *세이브 포인트* 다. 커밋은 *두 단계* 로 이뤄진다.

1. **스테이징 (Staging)** — 커밋할 파일들을 *고르는* 단계 (스테이징 영역에 올림)
2. **커밋 (Commit)** — 고른 파일들을 *메시지와 함께* 기록

**WebStorm** 에서는 **커밋 도구창** 으로 한다.

- **`Cmd + K`**(macOS) / **`Ctrl + K`**(Windows)로 커밋 창 열기
- 변경된 파일을 *체크* 해 스테이징하고, **커밋 메시지** 를 적은 뒤 **Commit** 클릭

```text
feat: 로그인 기능 구현
```

### 3. 커밋 메시지, 누가 쓰지? — 클로드 코드에게

강의에서는 *Cursor AI* 로 커밋 메시지를 자동 생성하지만, 우리에겐 더 좋은 카드가 있다. 바로 **클로드 코드** 다.

```text
지금 변경사항을 의미 있는 단위로 커밋해줘.
커밋 메시지는 컨벤션에 맞게 작성하고.
```

클로드 코드는 *변경 내용을 분석* 해 **적절한 커밋 메시지** 까지 붙여 커밋해준다. ([권한 섹션](/claude-code-클로드-코드-권한)에서 봤듯, `git` 명령은 *승인* 을 거치니 안심해도 된다.)
*터미널·GUI·클로드 코드* 중 **편한 방법** 을 고르면 된다.

> 💡 WebStorm 커밋 창에도 *AI Assistant* 의 **커밋 메시지 생성**(✨ 아이콘)이 있지만, [앞서](/claude-code-cursor-ai-ide-통합) 짚었듯 그건 *유료 별도 구독*
> 이다. *코딩도 커밋도*
> **클로드 코드** 에게 맡기는 게 이 시리즈의 일관된 길이다.

### 4. 커밋 전략 — "의미 있는 단위로"

여기서 *중요한 습관* 하나. **여러 기능을 한 번에** 커밋하면, 나중에 *특정 기능만 되돌리기* 가 어려워진다.

> 그러면 이러한 문제가 발생하지 않도록, 개발을 할 때는 **나만의 또는 우리 팀만의 커밋 전략** 이 필요한데요.

핵심은 **의미 있는 단위로 쪼개서** 커밋하는 것이다. *"로그인 기능"* 과 *"버그 수정"* 은 *따로* 커밋해야, 나중에 *한쪽만* 깔끔하게 롤백할 수 있다. *작고 명확한 커밋* 이 좋은 커밋이다.

### 5. 되돌아가기 — 체크아웃 (Checkout)

이력을 쌓아둔 *진짜 이유* 가 여기 있다. **이전 커밋 시점으로 되돌아가기**, 즉 **체크아웃 (Checkout)** 이다. *"회원가입 기능까지만 있던 그때"* 로 돌아가고 싶을 때 쓴다.

**WebStorm** 에서는 **Git 로그** 로 한다.

- 하단 **`Git` 도구창 → `Log` 탭** 에서 *커밋 히스토리* 확인
- 돌아갈 커밋을 **우클릭 → `Checkout Revision`**

그러면 *그 시점의 코드* 로 복원된다. *타임머신* 이라는 별명이 실감 나는 순간이다.

### 6. GitHub에 올리기 — 원격 저장소로

이제 *로컬* 프로젝트를 **GitHub (원격 저장소)** 에 올릴 차례다. **WebStorm** 이 이 과정을 매끄럽게 돕는다.

1. `Settings → Version Control → GitHub` 에서 **GitHub 계정 로그인**(최초 1회)
2. 상단 메뉴 **`Git → GitHub → Share Project on GitHub`**
3. 저장소 이름을 정하고, **Private**(비공개) 또는 **Public**(공개)을 선택해 게시

> GitHub에서 확인해 보면, 우리 프로젝트가 *원격 저장소인 GitHub에* **잘 올라온 걸** 확인할 수 있어요.

*이후의 변경* 은 **푸시 (Push)** 로 올린다. WebStorm에서는 **`Cmd + Shift + K`** / **`Ctrl + Shift + K`** 다. (물론 *"방금 커밋 GitHub에 올려줘"*
라고 **클로드 코드** 에게 시켜도 된다.)

### 7. 가져오기 — 클론 (Clone)

반대로, GitHub의 원격 저장소를 *내 PC로 통째로* 가져오는 게 **클론 (Clone)** 이다.

```bash
git clone <저장소 주소>    # 원격 저장소를 로컬로 복제
```

> 단순 *다운로드* 와 달리, 클론은 **소스 코드뿐 아니라 커밋 이력까지** 함께 가져온다.

**WebStorm** 에서는 *시작 화면* 의 **`Clone Repository`** 또는 **`Git → Clone…`** 으로 주소만 넣으면 된다. 클론 후 [
`webstorm .`](/claude-code-cursor-ai-ide-통합)
이나 WebStorm으로 열면, *작업하던 코드가 그대로* 세팅된 걸 볼 수 있다.

### 핵심 용어 정리

클로드 코드와 작업할 때 *용어를 알면* 의사소통이 훨씬 매끄럽다. 마지막으로 *복습* 하자.

| 용어                             | 의미                                 |
|----------------------------------|--------------------------------------|
| **작업 영역(Working Directory)** | 내가 *지금 편집 중* 인 파일들의 공간 |
| **스테이징 영역(Staging Area)**  | *커밋할 파일* 을 임시로 모아두는 곳  |
| **커밋(Commit)**                 | *특정 시점* 의 상태를 기록한 것      |
| **로컬 저장소(Local Repo)**      | *내 PC* 안의 Git 저장소(`.git`)      |
| **원격 저장소(Remote Repo)**     | *GitHub* 등 온라인의 저장소          |
| **체크아웃(Checkout)**           | *과거 커밋* 시점으로 코드를 되돌리기 |
| **클론(Clone)**                  | *원격 저장소* 를 이력째 로컬로 복제  |

### 정리하며

Git과 GitHub 기초를 정리하면 다음과 같다.

- **Git**(로컬 이력 관리) + **GitHub**(원격 백업·공유)
- **`git init`** → `.git` 생성 (WebStorm: `VCS → Enable Version Control Integration`)
- **커밋** = *스테이징 → 메시지와 함께 기록* (WebStorm: `Cmd/Ctrl + K`)
- **커밋 메시지·커밋 작업** → *클로드 코드* 에게 맡기면 편함
- **커밋 전략** → *의미 있는 단위* 로 쪼개야 부분 롤백 쉬움
- **체크아웃**(과거로) / **Share on GitHub**(업로드) / **클론**(가져오기)

처음엔 용어가 *낯설어도*, 강사님 말처럼 *쓰다 보면 자연스러워진다.* 게다가 우리는 **클로드 코드** 라는 든든한 짝이 있으니, *복잡한 Git 작업도* 말로 시키며 익혀가면 된다. 다음 챕터에서는 **Git
브랜치를 클로드 코드와 함께** 다루는, 한층 실전적인 워크플로우로 넘어가 보자.

## 🌿 Git 브랜치와 클로드 코드 활용

[앞 챕터](#-git-github-기초부터-실전까지)에서 *커밋·푸시·클론* 까지 익혔다. 이번엔 한 단계 위, **브랜치 (Branch)** 와 *실전 워크플로우* 다. **AI 시대** 엔 코드가 *폭발적으로*
쏟아지는데, 그럴수록 Git을 *제대로* 다루는 게 중요해진다. 핵심은 — *이 모든 걸* **클로드 코드** 에게 시킬 수 있다는 점이다.

> 말로 설명드리면 어려울 수 있기 때문에, *바로 보여드리도록* 할게요. **간단합니다.**

### Push & Pull — 로컬과 원격을 잇다

브랜치 전에, *로컬 ↔ 원격* 을 오가는 두 동작부터 정리하자.

| 동작     | 방향        | 설명                                      |
|----------|-------------|-------------------------------------------|
| **Push** | 로컬 → 원격 | *내가 커밋한 내용* 을 GitHub에 **올리기** |
| **Pull** | 원격 → 로컬 | *팀원이 올린 내용* 을 **내려받기**        |

혼자 작업해도 *백업* 을 위해 **Push** 하고, 협업 중이라면 *남의 변경* 을 받으려 **Pull** 한다. WebStorm에서는 **Push `Cmd/Ctrl + Shift + K`**, **Pull은
`Git → Pull`**(또는 *Update Project* `Cmd/Ctrl + T`)이다.

### 브랜치란 — "독립된 개발 라인"

이제 **브랜치** 다. 브랜치는 *독립적인 개발 라인* 이다. 왜 필요할까?

> 운영 중인 **메인 브랜치에서 직접 작업하는 것을 방지** 하기 위해, *개발용 브랜치를 별도로 분리* 해 진행해야 합니다.

생각해보자. *실제 사용자가 쓰는* `main` 브랜치에서 *바로* 코드를 고치다 *버그* 라도 나면? **서비스가 망가진다.** 그래서 **작업은 별도 브랜치** 에서 하고, *완성된 것만* `main`에 합치는
게 정석이다.

### 브랜치 전략 — main / develop / hotfix

대표적인 흐름은 **세 갈래** 다.

| 브랜치        | 역할                                              |
|---------------|---------------------------------------------------|
| **`main`**    | *실제 운영* 중인 안정 버전 (함부로 건드리지 않음) |
| **`develop`** | *새 기능 개발* 을 진행하는 작업 라인              |
| **`hotfix`**  | *운영 중 터진 긴급 버그* 를 급히 고치는 라인      |

평소엔 **`develop`** 에서 개발하다가, *완성되면* `main`에 **머지 (Merge)** 한다. 그런데 운영 중 *결제 오류* 같은 *급한 불* 이 나면? `main` 에서 곧장 **`hotfix`**
브랜치를 떼어 고치고, *바로* `main`에 머지해 배포한다. *개발 (develop)과 긴급수정 (hotfix)을 분리* 하니, 서로 *엉키지 않는다.*

> 신규 기능 구축보다 *기존 서비스를 안정적으로 유지관리* 하는 게 **더 어렵고 중요** 하며, 이를 위해 **브랜치 관리** 가 필수입니다.

### 클로드 코드로 브랜치 다루기

여기서 *이 시리즈의 묘미* 가 나온다. *복잡한 git 명령* 을 외울 필요 없이, **클로드 코드에게 말로** 시키면 된다.

**개발 브랜치 만들고 작업:**

```text
develop 브랜치를 새로 만들어서 체크아웃해줘.
그리고 방금 만든 기능을 커밋해줘.
```

**긴급 버그 — hotfix:**

```text
main 브랜치에서 hotfix 브랜치를 만들어줘.
결제 오류를 수정한 뒤 커밋하고.
```

**머지 (병합):**

```text
hotfix 브랜치를 main 브랜치에 머지하고, 원격(origin/main)에 푸시해줘.
```

강의에서도 *터미널에 `git merge` 를 직접 치는 대신* **클로드 코드로 머지** 를 시연한다. 클로드 코드가 *`main` 으로 체크아웃 → `hotfix` 병합 → 푸시* 까지 **알아서**
처리한다. 개발이 끝난 **`develop` → `main` 머지** 도 똑같이 *한마디* 면 된다.

```text
develop 브랜치를 main에 머지하고 푸시해줘.
```

> 그러면 이러한 개발이 완료됐다고 가정하고, `main` 브랜치에 머지를 해보겠습니다. **마찬가지로 Claude Code로 해볼게요.**

*Git 워크플로우의 거의 모든 동작* 을 이렇게 **클로드 코드로** 처리할 수 있다.

### WebStorm GUI로도 가능

물론 *눈으로 보며* 하고 싶다면 **WebStorm** 도 훌륭하다.

- **브랜치 생성·전환** → *우측 하단* **브랜치 위젯**(또는 `Git → Branches…`) → *New Branch* / *Checkout*
- **머지** → 브랜치 목록에서 합칠 브랜치 *우클릭 → `Merge into Current`*
- **브랜치 그래프** → 하단 **`Git` 도구창 → `Log` 탭** 에서 *갈라지고 합쳐지는* 흐름을 *시각적으로* 확인

특히 **Log 탭의 그래프** 는, *develop·main·hotfix* 가 어떻게 *분기·병합* 되는지 *한눈에* 보여줘 브랜치 전략을 이해하는 데 큰 도움이 된다. *클로드 코드로 작업하고,
WebStorm으로 확인* 하는 조합이 이상적이다.

### "용어를 알면 정말 좋다"

마지막으로 강사님이 *거듭* 강조한 포인트.

> 이래서 **용어를 알면 정말 좋다** 는 거예요. 이러한 용어는 **알고 계셔야 합니다.**

클로드 코드에게 *"hotfix 브랜치를 main에 머지해줘"* 라고 *정확히* 시키려면, **브랜치·머지·푸시** 같은 용어를 *알아야* 한다. [앞 챕터의 용어 정리](#-git-github-기초부터-실전까지)에
이번 것들을 더하면 이렇다.

| 용어       | 의미                                      |
|------------|-------------------------------------------|
| **Push**   | 로컬 커밋을 *원격에 올리기*               |
| **Pull**   | 원격 변경을 *로컬로 내려받기*             |
| **Branch** | *독립된 개발 라인*                        |
| **Merge**  | 한 브랜치의 변경을 *다른 브랜치에 합치기* |
| **Hotfix** | *운영 중 긴급 버그* 를 고치는 브랜치      |

### 정리하며

Git 브랜치와 클로드 코드 활용을 정리하면 다음과 같다.

- **Push**(올리기)· **Pull**(내려받기)로 로컬↔원격 동기화
- **브랜치** = 독립 개발 라인 → *`main` 직접 작업 금지*
- 전략 → **`main`**(운영)· **`develop`**(개발)· **`hotfix`**(긴급수정)
- *브랜치 생성·머지·푸시* 를 **클로드 코드에게 말로** 시킬 수 있음
- **WebStorm** 브랜치 위젯·`Log` 그래프로 *시각적 확인*
- **용어를 알아야** 클로드 코드에게 *정확히* 지시 가능

> 강의 내용이 *어렵게 느껴진다면*, 쉬운 부분만 이해하고 **나중에 다시 학습** 해도 좋습니다.

브랜치는 *처음엔 헷갈려도*, 클로드 코드와 *반복하다 보면* 자연스레 손에 익는다. 이것으로 **Git과 GitHub** 섹션을 마친다. 버전 관리라는 *든든한 안전망* 까지 갖췄으니, 이제 클로드 코드로 *마음껏
실험하고 되돌리며* 개발할 수 있다.

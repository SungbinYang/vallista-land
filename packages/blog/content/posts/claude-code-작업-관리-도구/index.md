## 이 강의가 추구하는 본질

본격적인 **작업 관리 도구** 를 배우기 전에, *반드시 짚고 가야 할* 이야기가 있다. **도구를 어떻게 바라볼 것인가** — 이 강의가 추구하는 *본질* 이다.

### 흔한 오해 — "AI가 알아서 다 해주겠지"

[복잡한 애플리케이션](/claude-code-notion-기반-온라인-견적서-시작하기)을 만들려면, *커다란 작업* 을 **작은 단위로 쪼개** 순차적으로 완료해야 한다. **Claude Task Master**
나 **Shrimp Task Manager** 같은 *작업 관리 도구* 가 이 과정을 돕는다. 그런데 여기서 *오해* 가 생긴다.

> 작업 관리 도구가 *알아서 작업을 나눠주고*, 순서대로 진행하면 **완벽한 앱이 나오겠지** — 라고 생각하기 쉬워요.

> 복잡한 판단은 *AI가 다 알아서 해주겠지.*

**아니다.** 이게 이 챕터가 깨려는 *첫 번째 환상* 이다.

### 도구는 '도구일 뿐'이다

작업 관리 도구가 하는 일은 *명확히 정해져 있다.*

| 도구가 하는 일            | 도구가 못 하는 일                     |
|---------------------|--------------------------------|
| 큰 작업을 *작은 단위* 로 나누기 | *어떻게 나누는 게 효율적인지* **판단**       |
| 순차 실행을 *정리*         | *올바른 개발 순서* 를 **설계**           |
| 작업 상황 *추적*          | *팀 수준·비즈니스 우선순위* 등 **컨텍스트** 이해 |
| 반복 작업의 *컨텍스트* 유지    | 바뀐 계획에 맞춰 **유연하게 핸들링**         |

> 즉 다시 말해서, **도구는 작업 관리를 할 수 있도록 도와주는 도구일 뿐** 이라는 거예요.

*나누고 정리하고 추적하는* 건 도구가 한다. 하지만 **"무엇을, 어떤 순서로, 왜"** 는 *여전히 사람의 몫* 이다.

### 주도권은 개발자에게 — 개발 '순서'의 중요성

[MVP로 작게 시작하는 것](/claude-code-notion-기반-온라인-견적서-시작하기)도 좋지만, *그만큼 중요한 게* **올바른 개발 순서** 다. 강사는 *아파트 건설* 에 빗댄다.

> *화장실을 만들기 전에*, 공통으로 설치해야 할 **프레임(골격)** 작업을 먼저 진행해야 효율적이죠.

소프트웨어도 똑같다. **애플리케이션 골격을 세우고 → 공통을 만든 뒤 → 개별 기능** 으로 가는 *순서* 가 성패를 가른다. 그리고 이 *순서 판단* 은 — 도구가 아니라 **개발자** 가 한다.

### 도구가 모르는 것 — 프로젝트 컨텍스트

순서만큼 중요한 게 **프로젝트의 컨텍스트(배경지식)** 다. *팀의 기술 수준*, *비즈니스 우선순위* 같은 건 **도구가 알 수 없다.** 오직 *개발자만* 안다.

그래서 작업 관리 도구를 쓰더라도, **개발자가 컨텍스트를 쥐고** 핸들링해야 한다. *계획은 언제든 바뀔 수 있고*, 그 변화에 맞춰 **유연하게 조정** 하는 것도 사람의 일이다.

### 핵심 — AI는 '0'을 '100'으로 만들지 못한다

이 챕터의 결론은 *하나의 문장* 으로 모인다.

> AI 도구는 우리의 **퍼포먼스를 늘려주는** 도구일 뿐이에요. 개발자가 **'0'이라면**, AI 도구의 퍼포먼스도 *발휘될 수 없죠.*

AI는 **곱셈** 이다. *내 역량에 곱해져* 생산성을 키운다. 하지만 *0에 무엇을 곱해도 0* 이다. 그러니 [이 강의의 진짜 목표](/claude-code-notion-기반-온라인-견적서-시작하기)도 *도구
사용법* 이 아니라 — **주도적으로 도구를 부리는 개발자** 가 되는 것이다.

> AI 기반 작업 관리 도구를 쓰더라도, *보다 더 잘 사용한다면* **복잡한 애플리케이션도 성공적으로** 만들 수 있어요.

### 정리하며

이 강의가 추구하는 본질을 정리하면 다음과 같다.

- **오해** → *"AI가 알아서 다 해주겠지"* → **환상이다**
- **도구의 역할** → *나누기·정리·추적* 까지 → **판단·순서·컨텍스트는 사람의 몫**
- **개발 순서** → *아파트 골격* 처럼 → **공통 먼저, 개별 기능 나중**
- **컨텍스트** → *팀 수준·우선순위* 는 도구가 모름 → **개발자가 핸들링**
- ⭐ **본질** → AI는 **곱셈** → *내가 0이면 결과도 0* → **주도권은 개발자에게**

작업 관리 도구는 *강력한 조수* 지만, **운전대는 내가 잡는다.** 이 마음가짐을 깔고 — 다음 챕터부터 **Task Master** 와 **Shrimp Task Manager** 를 *언제·어떻게* 쓰면 좋을지
구체적으로 들여다보자.

## 작업관리 도구 (Task Manager)

[앞 챕터](#이-강의가-추구하는-본질)에서 *마음가짐* 을 다졌으니, 이제 **실제 도구** 를 만날 차례다. 강사가 콕 집은 두 가지 — **Taskmaster AI** 와 **Shrimp Task Manager
** 다.

### 왜 필요한가 — "이런 고민, 해보셨나요?"

> 여러분 혹시 AI 도구를 사용하면서, *이런 고민* 해보신 적 있나요? **당근마켓이나 쇼핑몰** 같은 복잡한 애플리케이션은 *어떻게 체계적으로* 작업을 관리하고 개발하지?

*간단한 앱* 은 [계획 없이도](/claude-code-notion-기반-온라인-견적서-시작하기) AI로 뚝딱 만든다. 하지만 **규모가 커지면** 이야기가 달라진다.

> *커다란 문제* 를 **작업 단위로 쪼갠** 다음에, 하나하나씩 **순차적으로 완료** 해야만 하는데요.

이 *쪼개고 → 순서대로 → 추적하는* 과정을 도와주는 게 바로 **작업 관리 도구** 다.

### 두 도구 — Taskmaster AI & Shrimp Task Manager

| 도구                      | 한 줄 소개                                      |
|-------------------------|---------------------------------------------|
| **Taskmaster AI**       | AI 기반 워크플로우에서 *복잡한 앱* 의 **체계적 작업 관리** 지원    |
| **Shrimp Task Manager** | *큰 작업을 분해* 하고 **세션 간 컨텍스트** 를 유지하는 작업 관리 도구 |

둘 다 **커다란 문제를 작은 작업으로 나누고**, 그 작업을 *순차적으로 완료* 하도록 돕는다는 점에서 결이 같다. 강사의 한마디가 모든 걸 요약한다.

> 정말 유용합니다.

### 먼저 보고 오면 좋은 것 — 유튜브 영상

한 가지 안내가 있다. 두 도구의 *자세한 설치·사용법* 은 **유튜브에 별도로** 올라가 있다.

> 본 정규 강좌를 수행하기 전에, 해당 도구들의 **설치 및 사용법 영상** 을 먼저 시청하시는 걸 권장드려요.

즉 이 강의는 *설치 튜토리얼* 을 반복하지 않는다. 대신 **더 중요한 것** — *"언제 무엇을 쓸지"* 에 집중한다.

### 이 챕터의 진짜 주제 — '비교'

많은 학습자가 *가장 궁금해한* 게 바로 이거다.

> 이번 강의에서는 Taskmaster AI와 Shrimp Task Manager를 **언제 사용하면 좋은지** 제가 *비교해 드리도록* 하겠습니다.

[도구는 도구일 뿐](#이-강의가-추구하는-본질)이라는 본질을 기억하자. *어떤 도구가 더 좋다* 가 아니라, **내 프로젝트 상황에 어떤 도구가 맞는지** 를 *주도적으로 판단* 하는 게 핵심이다.

### 정리하며

작업 관리 도구를 정리하면 다음과 같다.

- **언제 필요?** → *간단한 앱* 은 불필요 → **당근마켓·쇼핑몰급 복잡한 앱** 에서 필수
- **무엇을?** → 큰 문제를 *작은 작업* 으로 쪼개 → **순차 완료·추적**
- **두 도구** → **Taskmaster AI** · **Shrimp Task Manager**
- **사용법** → *유튜브 영상* 으로 선행 학습 권장
- **이 챕터** → 설치가 아니라 **"언제 무엇을 쓸지" 비교** 에 초점

복잡한 프로젝트일수록 *체계적인 작업 관리* 가 성패를 가른다. 다음 챕터에서는 **Taskmaster vs Shrimp** — *두 도구를 직접 비교* 하며, *언제 어느 쪽* 을 집어 들면 좋을지 가려보자.

## Claude Task Master, Shrimp Task Manager

[앞 챕터](#작업관리-도구-task-manager)에서 *두 도구의 이름* 을 만났다. 이번 챕터는 그 둘 — **Claude Task Master** 와 **Shrimp Task Manager** 를 *제대로
들여다보기* 위한 **연결 고리** 다.

### 자세한 사용법은 유튜브에

강사는 이 두 도구를 *유튜브 영상* 에서 이미 **상세히** 다뤘다.

> 이번 시간에는 제가 *유튜브에 올렸던* **Claude Task Master와 Shrimp Task Manager** 에 대해 잠시 공유하도록 할게요. 수강생분들이 *꼭 한번 보셨으면* 하는 마음에 소개하게
> 되었습니다.

즉 이 챕터의 핵심 안내는 **"먼저 영상부터 보고 오세요"** 다. [설치·사용법을 정규 강좌에서 반복하지 않는다](#작업관리-도구-task-manager)는 흐름의 연장선이다.

### 영상 볼 때 한 가지만 — 호흡이 빠릅니다

> 유튜브 영상은 *컨셉에 맞게 제작* 되어 **호흡이 다소 빠를 수** 있어요. 이 점 *양해 부탁* 드립니다.

빠르게 지나가는 부분은 **멈춰가며·돌려가며** 보면 된다. 중요한 건 *완급* 이 아니라, 두 도구가 **어떻게 작업을 쪼개고 관리하는지** 그 *감* 을 잡는 것이다.

### 다음은 '비교' — 그리고 Q&A

이 챕터의 진짜 목적지는 **다음 챕터** 다.

> 다음 시간에는 이 두 도구를 **비교하는 시간** 을 가질 예정이에요.

그리고 강사는 *학습 경로* 까지 친절히 일러둔다.

| 단계          | 무엇을                                     |
|-------------|-----------------------------------------|
| **① 지금**    | 두 도구 *유튜브 영상* 시청                        |
| **② 다음 챕터** | Task Master vs Shrimp **비교 분석**         |
| **③ 이후**    | 궁금한 점은 **Q&A 게시판** → *유용한 질문은 추가 영상* 으로 |

> 비교 영상 시청 후 *궁금한 점* 은 **Q&A 게시판** 을 통해 질문해 주시면, 제가 *최대한 빠르게* 답변 드리겠습니다.

### 정리하며

이 챕터를 정리하면 다음과 같다.

- **성격** → 두 도구를 *제대로 보기 위한* **연결 고리** 챕터
- **할 일** → **Claude Task Master · Shrimp Task Manager** *유튜브 영상* 꼭 시청
- **유의** → 영상 *호흡이 빠를 수* 있음 → **돌려보며** 감 잡기
- **다음** → [다음 챕터](#작업관리-도구-task-manager)에서 **두 도구 비교** → 이후 **Q&A** 로 심화

[작업 관리 도구의 본질](#이-강의가-추구하는-본질)을 마음에 둔 채, 두 영상으로 *기본기* 를 채워두자. 다음 챕터에서 **Task Master와 Shrimp를 나란히 놓고** — *언제 어느 쪽* 이 맞는지
본격적으로 비교한다.

## 작업관리 도구 비교 (TaskMaster vs Shrimp)

[앞 챕터](#claude-task-master-shrimp-task-manager)에서 예고한 **비교** 의 시간이다. *어느 게 더 좋다* 가 아니라, **내 상황에 어느 쪽이 맞는지** 를 가리는 게 목표다.

### ① GitHub 인기도 — 객관적 지표부터

오픈소스를 고를 땐 **GitHub 통계** 가 좋은 출발점이다.

| 지표    | **Claude Task Master** | **Shrimp Task Manager** |
|-------|------------------------|-------------------------|
| ⭐ 스타  | **22,000**             | 1,800                   |
| 🍴 포크 | **2,200**              | 200                     |

> 오픈소스 프로젝트를 선택할 때 *GitHub 인기도* 는 정말 **중요한 지표** 죠.

숫자만 보면 **Task Master** 가 압도적이다. 하지만 강사는 *못* 을 박는다.

> 그렇다고 해서 *Shrimp Task Manager가 좋지 않다* — **이건 아니에요.**

Shrimp는 *신생 프로젝트* 임을 감안하면 **훌륭한 성과** 다. 인기도는 *참고* 일 뿐, **선택의 전부** 가 아니다.

### ② 신규 프로젝트 — 문서가 있나, 없나

가장 큰 갈림길은 **"개발 문서가 얼마나 탄탄한가"** 다.

| 상황                                                        | 추천 도구                   |
|-----------------------------------------------------------|-------------------------|
| *상세한 [PRD](/claude-code-notion-기반-온라인-견적서-시작하기)·TRD* 가 있다 | **Claude Task Master**  |
| *문서 없이* 빠르게 시작하고 싶다                                       | **Shrimp Task Manager** |

**Task Master** 는 *잘 작성된 PRD* 를 기반으로 **체계적인 작업 관리** 를 한다. 반대로 **Shrimp** 는 `init_project`·`plan_task` 기능으로 *"이커머스 플랫폼 개발
계획 세워줘"* 같은 **한 줄 프롬프트** 로 가볍게 출발한다.

> 문서가 부족한 상태에서 Task Master를 쓰면, 작업 계획이 **모범 사례가 아닐 수** 있어요.

[개발 순서가 중요하다](#이-강의가-추구하는-본질)는 본질과 같은 맥락이다 — *건물을 짓듯* **기반부터** 잘 세워야 하는데, 그 기반이 *바로 문서* 다.

### ③ 협업이냐, 유연함이냐

목표에 따라서도 갈린다.

- **Claude Task Master** → *팀 협업*, *클라이언트 요구사항 명확*, *문서 명확* → **체계적 관리**
- **Shrimp Task Manager** → *만들면서 개선*, *AI와 대화하며* → **유연한 방향 전환**

Task Master는 *디테일한 분석* 에 강해 **개발 리더** 가 팀에 업무를 나눌 때 빛난다. Shrimp는 *빠른 프로토타이핑* 에 강하다. 실제로 강사도 이렇게 말한다.

> 저는 현재 **1인 개발자** 로서 *빠른 결과* 를 선호하기 때문에, **Shrimp Task Manager** 를 유용하게 쓰고 있어요.

### ④ 프로젝트 중간 도입 — 진행 상황부터 분석

이미 *굴러가던 프로젝트* 에 도구를 넣는다면? 먼저 **"어디까지 왔고, 어떤 문제가 있는지"** 를 분석해야 한다.

> AI는 *우리 실력의 퍼포먼스를 배로* 늘려주는 것이지, **아무것도 안 했을 때** 키워주는 게 아니에요.

[이 강의의 본질](#이-강의가-추구하는-본질)이 또 등장한다. 중간 도입의 갈림길은 이렇다.

- **Task Master** → *현재 요구사항·진행 상황* 을 담은 **PRD를 정확히** 쓸 수 있다면 좋은 선택
- **Shrimp** → *가이드 문서를 쓰기 어렵다면* → **PRD 없이** `plan_task` 로 **부분적** 작업 관리

### ⑤ 메모리 관리 — Shrimp의 '지능형 메모리'

두 도구 모두 작업을 **`tasks.json`** 으로 관리한다(`.taskmaster/` vs `Shrimp` 데이터 폴더). 차이는 **그 너머** 에 있다.

> Shrimp는 *태스크뿐만 아니라*, **AI와 나눴던 컨텍스트(맥락)** 까지 기억해요.

**Shrimp** 는 *별도 메모리 파일* 로 *과거 작업·분석·패턴* 까지 자동 관리하는 **지능형 메모리** 를 갖췄다. 그래서 *이전 맥락을 참조* 하며 점점 더 나은 작업 관리를 돕는다.

### 한눈에 — 언제 무엇을?

| 기준        | **Claude Task Master** | **Shrimp Task Manager** |
|-----------|------------------------|-------------------------|
| **문서**    | 상세한 PRD/TRD *필요*       | *없어도* OK                |
| **시작 속도** | 느리지만 *체계적*             | **빠른** 프로토타이핑           |
| **강점**    | 디테일 분석·**팀 협업**        | 유연한 방향 전환·**1인 개발**     |
| **메모리**   | `tasks.json`           | + **지능형 메모리**(맥락 기억)    |

> 잘 정돈된 *문서 기반* 으로 체계적으로 관리하고 싶다면 **Task Master**, 빠르게 *프로토타입* 을 만들며 유연하게 가고 싶다면 **Shrimp** 가 유리해요.

### 정리하며

두 도구 비교를 정리하면 다음과 같다.

- **인기도** → *Task Master* 가 압도적 → 하지만 **선택의 전부는 아님**
- **신규** → *문서 탄탄* → Task Master / *문서 없이 빠르게* → Shrimp
- **목표** → *팀 협업·체계* → Task Master / *유연·프로토타입* → Shrimp
- **중간 도입** → *진행 상황 분석* 이 먼저 → PRD 가능하면 Task Master, 아니면 Shrimp
- **메모리** → Shrimp의 **지능형 메모리**(맥락까지 기억)가 차별점

결국 *정답은 하나가 아니다.* [도구는 도구일 뿐](#이-강의가-추구하는-본질)이니, **내 프로젝트의 문서 수준·목표·팀 구성** 을 보고 *주도적으로* 고르면 된다. 작업 관리 도구 시리즈는 여기까지 — 이제
*고른 도구* 로 **실제 프로젝트** 를 굴려볼 차례다.

## Supabase란 무엇인가? 사용하는 이유!

[지난 시리즈](/claude-code-관리자-기능-확장)에서 관리자 인증을 *환경 변수 패스워드* 로 임시 처리하며, **제대로 된 인증은 추후 Supabase로** 가겠다고 미뤄뒀다. 이제 그 약속을 지킬
차례다. 그런데 코드를 짜기 전에 — *Supabase가 대체 뭐길래* 다들 쓰는지부터 짚고 가자.

> 쉽게 말해서 Supabase를 사용하면 웹이나 모바일 애플리케이션을 만들 때 필요한 **백엔드 기능들** 을 *정말 쉽게* 구현할 수 있도록 도와줘요.

### 한 줄 정의 — 백엔드를 '빌려 쓴다'

웹앱을 만들려면 *프론트엔드(화면)* 만으론 부족하다. **회원가입·로그인** 을 처리하고, *데이터를 저장* 하고, *파일을 보관* 할 **백엔드** 가 필요하다. 문제는 — 이걸 *직접 다 만들면* **시간이
어마어마하게** 든다.

```text
[직접 구축]  서버 세팅 + DB 설치 + 인증 구현 + 파일 저장소 + …  → 며칠~몇 주
[Supabase]   이미 다 준비됨 → 가져다 연결만                      → 몇 시간
```

**Supabase** 는 이 *백엔드 기능 세트* 를 **미리 만들어 빌려주는** 서비스다. 흔히 **BaaS(Backend as a Service)** 라 부른다.

### 무엇을 주나 — 5가지 핵심 기능

강사가 *일상 서비스에 빗대* 하나씩 풀어준다.

| 기능                 | 무엇을                                 | 비유                  |
|--------------------|-------------------------------------|---------------------|
| **데이터베이스**         | *상품·주문·고객* 데이터를 **PostgreSQL** 에 저장 | 온라인 쇼핑몰의 상품/주문 정보   |
| **인증(Auth)**       | *이메일·소셜(구글/카카오)* 로그인 + **RLS**      | 로그인한 사람만 *제 데이터* 접근 |
| **실시간(Realtime)**  | *변경이 즉시* 다른 사용자에게 동기화               | 채팅 메시지가 바로 뜨는 것     |
| **스토리지(Storage)**  | *사진·동영상* 대용량 파일 저장                  | 인스타그램 업로드           |
| **Edge Functions** | *서버 없이* 커스텀 로직 실행                   | 결제 후 영수증 메일·썸네일 생성  |

여기서 특히 강조되는 게 **Row Level Security(RLS)** 다.

> 이것이 바로 Supabase의 **핵심 기능** 이라고 할 수 있죠.

RLS는 *각 사용자가 자신의 데이터에만* 접근하도록 **보안을 자동 처리** 한다. *남의 주문 내역* 이 보이는 사고를 — DB 레벨에서
막아준다. [지난 시리즈의 환경 변수 패스워드](/claude-code-관리자-기능-확장)와는 차원이 다른 *진짜 인증* 이다.

### 왜 쓰나 — 네 가지 이유

기능이 많다고 다 쓰는 건 아니다. *선택받는 이유* 는 따로 있다.

- **개발 속도** → 백엔드가 *이미 준비* 돼 있어 **비즈니스 로직·프론트** 에 집중. *MVP를 몇 시간 만에* 만들었다는 후기까지
- **비용 효율성** → *작은 프로젝트는 무료*, 성장하면 **쓴 만큼만** 지불 (직접 서버 운영보다 경제적)
- **높은 확장성** → 사용자가 *갑자기 100배* 늘어도 **스케일링을 알아서** 처리
- **뛰어난 개발자 경험(DX)** → *잘 정리된 문서* · 직관적 API · 쉬운 대시보드

여기에 *전 세계 개발자* 들이 정보를 나누는 **강력한 커뮤니티** 까지 더해진다. *막혔을 때* 검색하면 **답이 이미 있는** 경우가 많다는 뜻이다.

### Claude Code와 왜 잘 맞나

한 가지 덧붙이면 — Supabase는 [AI 워크플로우](/claude-code-mvp-개발-및-배포)와 *궁합이 좋다.* *기능이 표준화* 돼 있고 **문서가 탄탄** 하니, 클로드가 *적은 컨텍스트로도* 정확한
코드를 뽑는다. *백엔드를 손수 설계* 하는 것보다, **검증된 패턴** 을 AI가 엮어주는 게 훨씬 빠르고 안정적이다.

### 정리하며

Supabase를 정리하면 다음과 같다.

- **정체** → 백엔드 기능을 *미리 만들어 빌려주는* **BaaS**
- **5대 기능** → *DB(PostgreSQL) · 인증(+RLS) · 실시간 · 스토리지 · Edge Functions*
- ⭐ **RLS** → *각자 제 데이터만* 접근 → **보안 자동 처리**
- **선택 이유** → *개발 속도 · 비용 효율 · 확장성 · 개발자 경험* (+커뮤니티)
- **AI 궁합** → *표준화·문서화* → [Claude Code](/claude-code-mvp-개발-및-배포)가 적은 컨텍스트로 정확히

강사의 한마디가 Supabase의 본질을 요약한다.

> Supabase는 단순히 도구가 아니라, 우리의 *아이디어를 빠르게 현실로* 만들어주는 **강력한 파트너** 예요. *복잡한 백엔드 설정* 에 시간을 쓰지 않고, 정말 중요한 것 — **사용자에게 가치를 전달하는
것** 에 집중할 수 있게 해주죠.

개념을 잡았으니, 다음 챕터에서는 [Claude Code](/claude-code-mvp-개발-및-배포)와 함께 **Supabase 프로젝트를 직접 생성** 해 우리 앱에 연결해보자.

## Supabase 프로젝트 생성하기

[앞 챕터](#supabase란-무엇인가-사용하는-이유)에서 *Supabase가 무엇인지* 개념을 잡았다. 이제 **직접 프로젝트를 만들어** Next.js와 연결할 차례다. *데이터베이스 + 로그인·회원가입* 이
얽힌 — 이 시리즈에서 가장 *복잡한* 세팅이지만, 강사가 천천히 짚어준다.

> 경험이 적으신 분들도 *잘 따라오실 수 있도록* 요.

### 왜 'AI'가 아니라 '공식 문서'로 시작하나

여기서 강사가 *의외의 선택* 을 한다. *"Claude한테 Supabase 세팅해줘"* 라고 시키지 않는다. **공식 문서를 그대로 따라간다.**

> 초기 프로젝트를 구성할 때 *AI를 통해서* 할 수도 있지만, *공식 문서로 구성* 하는 건 **다양한 이점** 이 있다고 했죠?

이유는 둘이다. *세팅은 한 번뿐이고 정확도가 생명* 이라 — **검증된 공식 절차** 가 안전하고, 매번 *방대한 세팅을 AI에 설명* 하면 [토큰만 낭비](/claude-code-mcp-서버-설치)된다. *AI를
안 쓰는 게* 아니라, **쓸 곳에 쓴다** 는 분별이다.

> 첫 단추가 *정말 중요* 하거든요. (…) **아는 만큼 더 보이기** 때문이죠.

### ① 프로젝트 생성

[Supabase 공식 문서](https://supabase.com/docs)의 `Docs → Start` 에서 **Next.js** 를 고르면 연동 가이드가 나온다. `database.new` 로 새 프로젝트를
만든다.

- **프로젝트 명** → `nextjs-supabase-app`
- **DB 패스워드** → *직접 지정* (꼭 기억해둘 것)
- **Region** → **서울(Seoul)** — *사용자와 가까울수록* 응답이 빠르다

> 💡 **Region은 '데이터가 물리적으로 사는 곳'.** 데이터센터가 *멀수록* 왕복 시간이 길어진다. 국내 서비스라면 *서울 리전* 이 정답.

### ② 테이블 생성 + RLS — 문서 그대로 복붙

프로젝트가 생기면, 공식 문서의 *SQL을 그대로 복사* 해 대시보드의 **`SQL Editor`** 에 붙여 실행한다.

1. **`instruments` 테이블** 생성 + 샘플 데이터 삽입 (문서 제공 SQL)
2. **[RLS 정책](#supabase란-무엇인가-사용하는-이유)** 설정 — *익명 사용자(`anon`)* 에게 **`SELECT` 만 허용**

> 어렵지 않죠? **공식 문서 그대로** 따라 하고 있습니다.

2번이 [앞 챕터에서 강조한 RLS](#supabase란-무엇인가-사용하는-이유)의 실물이다. *로그인 안 한 방문자* 도 이 테이블은 **읽을 수 있게** 여는 정책이다.

### ③ Next.js 프로젝트 — `with-supabase` 템플릿

프론트는 *맨바닥부터* 깔지 않는다. **`with-supabase` 템플릿** 으로 시작한다.

```bash
npx create-next-app -e with-supabase nextjs-supabase-app
```

이 템플릿엔 **쿠키 기반 인증 · TypeScript · Tailwind CSS** 가 *미리 세팅* 돼 있다. 개발 서버를 띄우면 — [지난 프로젝트](/claude-code-mvp-개발-및-배포)와 달리
*Supabase가 통합된 UI* 가 바로 뜬다.

### ④ 환경 변수 + 클라이언트 연결

이제 *내 프로젝트* 와 *Supabase* 를 잇는다. `.env.example` 을 **`.env.local`** 로 바꾸고 두 값을 채운다.

| 환경 변수                               | 의미                       |
|-------------------------------------|--------------------------|
| **`NEXT_PUBLIC_SUPABASE_URL`**      | Supabase 프로젝트 **API 주소** |
| **`NEXT_PUBLIC_SUPABASE_ANON_KEY`** | 클라이언트용 **공개(anon) 키**    |

값은 *Supabase 프로젝트 설정* 에서 복사해 붙인다. 그다음 **`utils/supabase/server.ts`** 에 *클라이언트 생성 함수* 를, **`app/instruments/page.tsx`** 에
*데이터 조회 코드* 를 (역시 문서대로) 넣는다.

### ⑤ 동작 확인 — 그리고 흔한 함정 둘

환경 변수가 잡히면 *회원가입·로그인 버튼* 이 **활성화** 된다. 여기서 강사가 *실제로 부딪힌* 두 함정을 보여준다.

```text
[함정 1] 로그인 오류  → 서버 재실행 + 브라우저 쿠키 삭제 → 해결
[함정 2] 키명 불일치  → ANON_KEY 키명이 가이드와 다름 → 올바른 이름으로 수정
```

- **로그인이 안 될 때** → *쿠키가 쌓여서* 그런 경우가 많다. **서버 재실행 + 쿠키 삭제** 면 풀린다.
- **Instruments 페이지 오류** → 오류 로그를 보니 *키 이름* 이 가이드와 달랐다. `.env.local` 의 **`ANON_KEY` 키명을 수정** 하면 오류는 사라진다. (*목록이 아직 안 뜨는
  건* 다음 챕터에서 다룬다.)

[오류는 자연스러운 것](/claude-code-mvp-개발-및-배포) — 중요한 건 *로그를 읽고 고치는* 흐름이다.

### 정리하며

Supabase 프로젝트 생성을 정리하면 다음과 같다.

- **시작점** → *AI 말고* **공식 문서** → 정확도 ↑·[토큰](/claude-code-mcp-서버-설치) 절약 (쓸 곳에만 AI)
- **① 프로젝트** → `database.new` → 이름·DB 패스워드·**서울 리전**
- **② 테이블·RLS** → `SQL Editor`에 *문서 SQL 복붙* → `anon`에 **`SELECT` 허용**
- **③ Next.js** → `create-next-app -e with-supabase` (*인증·TS·Tailwind 내장*)
- **④ 연결** → `.env.local`에 **URL·ANON_KEY** → `utils/supabase` 클라이언트
- **⑤ 함정** → *로그인 오류=쿠키 삭제* · *오류=키명 수정*

세팅은 *지루하지만* **첫 단추** 다. 공식 문서는 처음엔 어려워도, *자주 보면 익숙* 해지고 **아는 만큼 더 보인다.** 백엔드 토대가 깔렸으니 — 다음
챕터에서는 [Claude Code](/claude-code-mvp-개발-및-배포)와 함께 이 Supabase 위에 **실제 기능** 을 올려보자.

## Supabase RLS 정책 이슈 해결

[앞 챕터](#supabase-프로젝트-생성하기) 끝에 *떡밥* 을 하나 남겼다. **오류는 사라졌는데 목록이 안 뜬다** 는 것. 분명 [공식 문서대로](#supabase-프로젝트-생성하기) *테이블 생성 → 더미
데이터 → RLS 정책* 까지 다 했는데 — *데이터가 조회되지 않는다.*

> 이번 시간에는 우리가 해당 페이지로 진입을 했을 때, (왜 데이터가 안 나오는지) **말씀드리도록 하겠습니다.**

### 증상 — 분명 다 했는데 비어 있다

상황을 정리하면 이렇다. `instruments` 테이블에 **`public can read instruments`** 라는 [RLS 정책](#supabase란-무엇인가-사용하는-이유)을 걸어뒀다. 이 정책은 *익명
사용자(`anon`)* 에게 읽기를 허용한다. *에러도 없고* 정책도 *분명 있는데* — 화면은 **텅** 비어 있다.

> 그런데 **보세요.**

### 원인 — 지금 나는 `anon`이 아니다

범인은 **롤(role)의 불일치** 다. `protected` 페이지는 *로그인을 해야* 들어간다. 즉 지금 나는 **로그인한 사용자** — 롤이 `anon`이 아니라 **`authenticated`** 다.

```text
정책 허용 대상:  anon (익명)            ← 읽기 OK
실제 내 롤:      authenticated (로그인됨) ← 정책에 해당 안 됨 → 조회 0건
```

*RLS는 정확히 일을 한 것* 이다. **`anon`에게만** 열어줬으니, **`authenticated`인 나는** *대상이 아니라* 막힌 것. *에러가 아니라* — **정책상 보여줄 게 없다** 가 정답이었다.

### 해결 — 정책을 `authenticated`까지 열기

> 어떻게 수정을 해야 되죠?

답은 간단하다. `instruments` 테이블의 RLS 정책을 **`authenticated` 사용자도 조회 가능** 하도록 고친다. 저장하고 *페이지를 새로고침* 하면 — **데이터가 정상적으로** 뜬다.

```text
[수정 전] anon 만 SELECT 허용
[수정 후] anon + authenticated SELECT 허용  → 목록 출력 ✅
```

### 핵심 교훈 — 데이터가 안 보이면 RLS부터 의심하라

이게 *Supabase 입문자가 가장 많이* 걸리는 함정이다. 강사가 **꼭 기억하라** 고 못 박는다.

> 개발 중 *데이터 조회가 안 될* 경우, 가장 먼저 **해당 테이블의 RLS 정책을 확인** 하는 것이 중요하다.

특히 이건 [AI에게 맡기기 까다로운](/claude-code-mvp-개발-및-배포) 종류의 문제다. *에러 로그가 안 뜨니* — 클로드도 *"코드는 멀쩡한데?"* 하며 헤맬 수 있다. **사람이 RLS를 짚어줘야
** 빨리 풀린다. *AI가 만능은 아니라는* 좋은 예다.

### 마무리 — 코드 정리와 커밋

문제를 잡았으니 *실습용 군더더기* 를 치운다.

- **중복 제거** → [템플릿에 이미 있는](#supabase-프로젝트-생성하기) *서버 클라이언트 함수* 를 쓰고, `utils`에 따로 만든 코드는 **삭제**
- **테스트 잔재 정리** → *샘플 조회 페이지* 와 `instruments` *테이블* 도 제거
- **커밋** → 소스 제어에서 *환경 변수 키 변경* 외 이상 없음 확인 → 스테이징 → **커밋**

### 정리하며

RLS 정책 이슈 해결을 정리하면 다음과 같다.

- **증상** → 테이블·데이터·정책 다 있는데 *목록이 빈다* (에러 없음)
- **원인** → 정책은 **`anon`** 허용 / 로그인한 나는 **`authenticated`** → *대상 불일치*
- **해결** → RLS 정책을 **`authenticated`까지** 열기 → 새로고침 → 출력
- ⭐ **교훈** → *데이터 안 보이면* → **RLS 정책부터 확인** ([AI도 잡기 어려운](/claude-code-mvp-개발-및-배포) 영역)
- **마무리** → *중복·테스트 잔재* 제거 → **커밋**

RLS는 *강력한 보안* 이지만, 그만큼 **롤을 정확히** 다뤄야 한다. 이 *함정 하나* 를 알고 나면, 앞으로 *데이터가 안 보일 때* 헤매는 시간이 확 줄어든다. Supabase 토대와 *대표적 함정* 까지
익혔으니 — 다음 시리즈에서는 [Claude Code](/claude-code-mvp-개발-및-배포)와 **Supabase MCP** 로 *진짜 인증·기능* 을 본격적으로 붙여보자.

## Supabase MCP 사용하기: 로그인 구현하기

[앞 챕터](#supabase-rls-정책-이슈-해결)에서 *RLS 함정* 까지 넘으며 Supabase 토대를 다졌다. 그런데 지금까지는 *대시보드를 손으로* 오가며 SQL을 복붙했다. 이제 **클로드가 직접 DB를
만지게** 할 차례다 — 그 다리가 **Supabase MCP** 다.

### LLM의 한계 — DB에 직접 손이 안 닿는다

[ChatGPT·Claude](/claude-code-mvp-개발-및-배포) 같은 LLM은 *똑똑하지만* — **내 데이터베이스를 직접 들여다볼 수는 없다.** 그래서 매번 *"테이블 구조가 이렇고…"* 라고 사람이
설명해줘야 했다. [MCP](/claude-code-mcp-서버-설치)가 이 벽을 허문다.

> 쉽게 말씀드리면, **Supabase MCP** 는 Claude와 같은 LLM과 *Supabase 프로젝트를 연결* 해 주는 **다리 역할** 이라고 보시면 돼요.

```text
[기존]  Claude  →  ❓  →  Supabase DB   (직접 못 봄, 사람이 설명)
[MCP]   Claude  ←→ MCP ←→ Supabase DB   (직접 조회·생성·수정)
```

### ⚠️ 먼저 — 보안부터

MCP는 *DB에 직접 손* 을 댄다. 강력한 만큼 **위험** 도 따른다.

> Supabase MCP를 활용해서 *우리 데이터베이스에 직접 접근* 할 수 있기 때문에, 아무래도 **보안이 중요** 하겠죠.

강사가 못 박는 **수칙** 은 이렇다.

- **개발 환경에서만** — *운영(production)* DB엔 절대 연결하지 않는다
- **고객에게 미제공** — *순수 개발 도구* 로만
- **`read-only` 모드** — 데이터 *수정이 불필요* 하면 읽기 전용으로
- **프로젝트 스코프 · Feature Group** — *최소 권한 원칙* (필요한 기능에만 접근 허용)

### 설치 — `.mcp.json`까지

Supabase Docs의 **MCP 메뉴** 에서 설정을 만든다.

1. **플랫폼** → Hosting 버전 선택 → *연결할 프로젝트* 지정
2. **권한** → 필요 시 **`read-only`** 활성화, **Feature Group**(DB·Auth·Storage 등)으로 *접근 범위* 제어
3. **클라이언트** → **Claude Code** 선택 → *CLI 명령어 복사*
4. 프로젝트 **루트에서 실행** → **`.mcp.json`** 설정 파일 생성

### 연결 — `/mcp`로 인증

[Claude Code](/claude-code-mvp-개발-및-배포)에서 **`/mcp`** 를 입력하면 Supabase MCP 서버가 보인다.

```text
/mcp → [Supabase: disconnected]
     → Enter → Authenticate (API 접근 권한 부여)
     → [Supabase: connected] ✅
     → View tools (제공 기능 확인)
```

`disconnected` 상태에서 **`Authenticate`** 로 권한을 주면 **`connected`** 로 바뀐다. `View tools` 로 *어떤 기능* 을 쓸 수 있는지 훑어볼 수 있다.

### 실전 — `profiles` 테이블을 말로 만든다

이제 *진짜* 다. 회원가입 사용자 정보를 담을 **`profiles` 테이블** 을 — *SQL을 짜지 않고* **요청만** 한다.

1. 클로드가 **`list_tables`** 로 *현재 DB 상태* 를 직접 확인
2. `profiles` 테이블 **생성 계획** 수립 (⭐ [**RLS 설정도 계획에 포함**](#supabase-rls-정책-이슈-해결))
3. *민감한 이메일·비밀번호* 는 일단 제외하고 최종안 작성
4. [`Bypass permissions`](/claude-code-클로드-코드-권한) 모드로 실행 → **대시보드에 테이블 생성 확인**

[지난 챕터에서 사람이 손수 짚어야 했던 RLS](#supabase-rls-정책-이슈-해결)를, 이번엔 **MCP가 계획 단계부터** 챙긴다는 게 포인트다.

### 자동 프로필 생성 트리거

더 놀라운 건 그다음이다. 회원가입을 해보면 — `profiles` 테이블에 사용자 정보가 **자동으로 등록** 돼 있다.

> Supabase MCP 서버에 의해서 생성된 **자동 프로필 생성 트리거** 가 동작을 한 거죠.

MCP가 *테이블만 만든 게 아니라*, **가입 시 프로필이 자동으로 꽂히는 트리거** 까지 깔아준 것이다. *SQL·트리거 지식이 부족해도* — **말로 요청** 하니 알아서 처리됐다.

### 정리하며

Supabase MCP 로그인 구현을 정리하면 다음과 같다.

- **왜** → LLM은 *DB에 직접 접근 불가* → **MCP가 다리** 역할
- ⚠️ **보안** → *개발 환경만 · 고객 미제공 · `read-only` · 최소 권한*
- **설치** → Docs에서 설정 → CLI 실행 → **`.mcp.json`** 생성
- **연결** → [`/mcp`](/claude-code-mcp-서버-설치) → **Authenticate** → `connected`
- **실전** → `list_tables`로 상태 확인 → `profiles` 생성([RLS 포함](#supabase-rls-정책-이슈-해결)) → [`Bypass`](/claude-code-클로드-코드-권한)
- ⭐ **자동화** → *가입 시* **프로필 자동 등록 트리거** 까지 MCP가 구성

강사의 마무리가 이 기능의 의미를 압축한다.

> Supabase MCP 서버와 프롬프트를 잘 활용하면, **데이터베이스나 SQL 지식이 부족하더라도** Supabase의 다양한 서비스들을 *효과적으로 활용* 할 수 있다.

이제 *대시보드를 손으로* 오갈 필요가 줄었다. [클로드가 DB를 직접 보고 만지니](/claude-code-mvp-개발-및-배포) — *테이블 설계·RLS·트리거* 까지 **대화로** 풀린다. 이 다리를 놓았으니,
앞으로의 기능 구현은 *훨씬 빠르고 정확* 해질 것이다.

## Supabase 프로젝트 Claude 초기화

[앞 챕터](#supabase-mcp-사용하기-로그인-구현하기)에서 *Supabase MCP* 라는 다리를 놓았다. 이제 **본격 개발에 앞서** — 새 Supabase 프로젝트에 *그동안 갈고닦은 무기* 를
한꺼번에 장착할 차례다. [지난 견적서 StarterKit](/claude-code-mvp-개발-및-배포)에 깔아둔 *MCP·서브 에이전트·커맨드·훅* 을 **그대로 이식** 하는 작업이다.

> 이전 시간에 했던 내용이기 때문에 *조금 빠르게* 진행하도록 할게요.

### 왜 '이식'인가 — 매번 새로 만들지 않는다

[StarterKit의 진가](/claude-code-mvp-개발-및-배포)가 여기서 드러난다. *프로젝트마다* MCP를 새로 붙이고 에이전트를 다시 짜면 — **시간 낭비** 다. 한 번 잘 만들어둔 *세팅 묶음*
을 **복사해 가져오면**, 새 프로젝트도 *곧장 풀무장* 상태로 출발한다.

```text
[견적서 StarterKit]              [Supabase 앱]
 .mcp.json (Playwright~shrimp) ──복사──▶ 그대로 적용
 agents/ · commands/ · hooks/  ──복사──▶ 그대로 적용
 docs/guides/                  ──복사──▶ 그대로 적용
```

### ① MCP 서버 이식

먼저 [MCP 설정](/claude-code-mcp-서버-설치)부터다. `.mcp.json` 의 **[Playwright MCP](/claude-code-mcp-서버-설치)
부터 [shrimp-task-manager](/claude-code-shrimp-task-manager-도입) 까지** 전부 복사해 Supabase 앱에 붙인다.

> 💡 **`shrimp` 경로 주의.** 설정 하단 `shrimp` 부분에 *예전 `invoice-web`* 경로가 박혀 있다. 이걸 **현재 디렉터리** 로 바꿔줘야 한다. *경로를 안 고치면* 엉뚱한 프로젝트의
> 작업을 바라본다.

붙였으면 터미널에서 확인한다.

```bash
claude mcp list   # 설치된 MCP 목록 확인
```

[Supabase 인증](#supabase-mcp-사용하기-로그인-구현하기)이 필요하면 **`/mcp`** 로 *Authenticate* 를 거친다. 모든 서버가 **`connected`** 면 이식 성공이다.

### ② 컨텍스트·에이전트·커맨드·훅 이식

다음은 *개발에 쓸 자산* 들이다. StarterKit에서 **통째로 복사** 해 온다.

- **`docs/guides/`** → [Claude Code 컨텍스트 추가](/claude-code-설정-파일과-메모리-관리) 관련 *개발 가이드 문서*
- **`agents/`** → 그동안 만든 [서브 에이전트](/claude-code-클로드-코드-고급-서브-에이전트)들
- **커스텀 커맨드 · [훅](/claude-code-클로드-코드-고급-훅)** → 자주 쓰는 *자동화 세트*

루트 `docs` 아래 `guides`에 가이드를 넣고, 나머지도 *제자리* 에 붙이면 — *새 프로젝트인데도* **익숙한 작업 환경** 이 통째로 따라온다.

### ③ `/init` — `CLAUDE.md` 생성

세팅이 끝났으면 **프로젝트 초기화** 다. 터미널에서 Claude Code를 열고 **`/init`** 을 실행한다.

> 그러면 Claude Code가 **프로젝트의 전체 코드베이스를 스캔** 해서 이를 참조할 수 있는 **`CLAUDE.md`** 파일을 만들죠.

[`CLAUDE.md`](/claude-code-설정-파일과-메모리-관리)는 *클로드가 늘 참조하는* **프로젝트 설명서** 다. 생성된 파일을 보면 — *Next.js + Supabase 스타터 킷* 임을 소개하고
**주요 기술 스택** 을 정리해둔 게 보인다. 이걸 깔아두면, 이후 모든 작업에서 클로드가 *프로젝트 맥락을* 자동으로 안고 간다.

### ④ 커밋

끝으로 [Git 커밋](/claude-code-mvp-개발-및-배포)으로 *세팅 완료 시점* 을 박아둔다.

> 이러한 것들도 *커밋을 하시겠습니까?* 라고 묻고 있는데요, **모두 커밋** 해주세요.

### 정리하며

Supabase 프로젝트 Claude 초기화를 정리하면 다음과 같다.

- **컨셉** → *매번 새로 X* → [StarterKit](/claude-code-mvp-개발-및-배포) 세팅을 **복사·이식**
- **① MCP** → `.mcp.json` 전체 복사 → ⚠️ **`shrimp` 경로 수정** → `claude mcp list`·[`/mcp`](#supabase-mcp-사용하기-로그인-구현하기) →
  `connected`
- **② 자산** → [`docs/guides`](/claude-code-설정-파일과-메모리-관리)·[`agents`](/claude-code-클로드-코드-고급-서브-에이전트)
  ·커맨드·[훅](/claude-code-클로드-코드-고급-훅) 통째 이식
- **③ 초기화** → [`/init`](/claude-code-설정-파일과-메모리-관리) → 코드베이스 스캔 → **`CLAUDE.md`** 생성
- **④ 커밋** → *모두* 커밋

좋은 도구는 *한 번 만들어 계속 재사용* 한다. 이렇게 **무장한 프로젝트** 위에서라야 [AI 워크플로우](/claude-code-mvp-개발-및-배포)가 제 속도를 낸다. 토대와 장비를 모두 갖췄으니 — 다음
시간엔 AI 개발의 *오류를 줄여줄* **린트 체크·타입 체크** 를 더해, 더 단단한 개발 환경을 만들어보자.

## 개발도구 설정: Linter, TypeCheck

[앞 챕터](#supabase-프로젝트-claude-초기화)에서 *세팅을 이식* 하고 [`CLAUDE.md`](/claude-code-설정-파일과-메모리-관리)까지 깔며 **무장한 프로젝트** 를 만들었다. 마지막
한 겹이 남았다 — **개발 도구** 다. *ESLint·Prettier·타입 체크* 를 붙여, [AI 개발의 오류를 입구에서](/claude-code-mvp-개발-및-배포) 막는다.

> 이번에는 *ESLint나 프리티어 타입 체크* 와 같은 **개발 도구를 설정** 하도록 할게요.

### 왜 깔까 — AI가 짠 코드일수록 더

이 도구들은 *코드 품질의 그물망* 이다.

- **ESLint** → *문법·스타일 오류* 를 사전에 잡는 **린터**
- **Prettier** → *코드 포맷* 을 자동 정렬 (협업 시 통일성)
- **타입 체크** → *타입 불일치* 를 컴파일 전에 검출

[AI는 코드를 한꺼번에 많이 쏟아낸다](/claude-code-mvp-개발-및-배포). 그만큼 *사소한 실수* 가 끼어들 틈도 많다. 이 그물망이 깔려 있으면 — **클로드가 제 코드를 스스로 검증** 하며
*오류를 줄인다.*

### 핵심 — 프롬프트를 '전문가'로 시작한다

설정도 [클로드에게 맡긴다](/claude-code-mvp-개발-및-배포). 다만 *던지듯* 시키지 않는다. 강사의 프롬프트 작법이 핵심이다.

> **당신은 웹 애플리케이션 개발 도구 설정 전문가입니다.**

이렇게 *역할(롤)을 부여* 하고 — *원하는 도구를 간략히* 적되, **추천할 게 있으면 추천해달라** 고 여지를 둔다. 그리고 *두 가지 사고 기법* 을 얹는다.

- **COT(Chain-of-Thought)** → *단계별로 생각* 하며 진행
- **[`ultrathink`](/claude-code-mvp-개발-및-배포)** → **현재 프로젝트 구조를 깊이 분석** 해 *최적화된 설정* 도출

핵심은 **"우리 프로젝트에 맞게"** 다. *일반론적 설정* 이 아니라, [지금 이 코드베이스](#supabase-프로젝트-claude-초기화)를 분석한 **맞춤 설정** 을 받아낸다.

### 그리고 — 오류를 줄이는 두 무기

여기서 [MVP 때 배운 그 원칙](/claude-code-mvp-개발-및-배포)이 다시 등장한다.

> 오류를 *크게 줄일 수 있는* 방법이 **두 가지** 가 있다고요.

```text
① Playwright MCP   → 클로드가 스스로 화면 테스트·버그 수정
② 작업 완료 체크리스트 → 구현 후 lint·type·build 통과해야 '완료'
```

이번 설정의 목적이 바로 *②번 체크리스트* 를 **실제로 굴러가게** 만드는 것이다. 린트·타입 체크가 [`package.json`](/claude-code-mvp-개발-및-배포)에 박혀야, *"구현 → 자가
검증"* 루프가 완성된다.

### 작업 계획 — 그리고 실행

[`plan`](/claude-code-클로드-코드-권한)으로 잡힌 단계는 대략 이렇다.

1. **패키지 설치** — ESLint·Prettier·타입 관련 의존성
2. **Prettier 설정 파일** 생성
3. **ESLint 설정** 업데이트
4. **TypeScript 설정** + **`package.json` 스크립트** 추가 (`check-all` 류)
5. [**`CLAUDE.md` 업데이트**](/claude-code-설정-파일과-메모리-관리) — *개발 도구 사용법* 을 컨텍스트에 명시

계획을 확인했으면 [`Bypass permissions`](/claude-code-클로드-코드-권한) 모드로 *쭉* 진행한다.

> 이대로 **바이패스 모드** 로 진행하도록 할게요.

5번이 *은근히 중요* 하다. [`CLAUDE.md`에 "구현 후 `check-all`을 돌려라"](/claude-code-설정-파일과-메모리-관리)고 적어두면 — *이후 모든 작업* 에서 클로드가 **자동으로 자가
검증** 한다.

### 정리하며

개발 도구 설정을 정리하면 다음과 같다.

- **무엇** → **ESLint**(린트) · **Prettier**(포맷) · **타입 체크** → 코드 품질 그물망
- **왜** → [AI가 쏟아낸 코드](/claude-code-mvp-개발-및-배포)의 *틈* 을 자동으로 메움
- **프롬프트** → *"개발 도구 설정 전문가"* 롤 + **COT·[`ultrathink`](/claude-code-mvp-개발-및-배포)** → 맞춤 설정
- **오류 감소 2무기** → [Playwright MCP](/claude-code-mcp-서버-설치) + **작업 완료 체크리스트**
- **계획** → *설치 → 설정 파일 → 스크립트 → [`CLAUDE.md`](/claude-code-설정-파일과-메모리-관리)* → [`Bypass`](/claude-code-클로드-코드-권한)

이로써 **Supabase 어플리케이션의 토대** 가 완성됐다. *백엔드(Supabase)* · *MCP 다리* · *이식한 무기* · *코드 품질 그물망* 까지 — **개발을 시작할 모든 준비** 가 끝났다.
돌아보면 이 시리즈는 *기능 하나* 가 아니라, **단단한 출발선** 을 만드는 과정이었다. 잘 깔린 기반 위에서라면 — 앞으로의 *진짜 기능 개발*
은 [그 익숙한 워크플로우](/claude-code-mvp-개발-및-배포)로 *거침없이* 나아갈 수 있다.

## 서브에이전트활용: 구글 로그인 기능 추가하기

[앞 챕터](#개발도구-설정-linter-typecheck)까지 *토대* 를 모두 깔았다. 이제 **첫 실전 기능** 을 붙인다 — 거의 모든 서비스에 들어가는 **구글 로그인(OAuth)** 이다. 그런데 강사는
*기능부터 짜지* 않는다. **전문가 서브 에이전트** 를 먼저 세운다.

### 1단계 — `Next.js + Supabase 풀스택` 에이전트

> 제 생각에는 *이러한 Sub-agent가 있으면* 정말 좋을 것 같거든요. 그렇기 때문에…

[서브 에이전트](/claude-code-클로드-코드-고급-서브-에이전트)는 *기술 영역을 전문화* 한 AI다. 이번엔 **Next.js + Supabase 풀스택 개발자** 를 만든다. 단순히 이름만 붙이는 게
아니라 — [컨텍스트를 두툼하게](/claude-code-mvp-개발-및-배포) 채운다.

- **참고 자료** → *Next.js 최신 가이드* · *Supabase MCP 개발 가이드* → **모범 사례** 적용
- **사고 예산** → [`ultrathink`](/claude-code-mvp-개발-및-배포)로 *더 깊이* 분석
- **룰** → **보안 · 성능 최적화 · 에러 처리** 등 전문성 강화

> **첫 단추가 중요** 하기 때문에 *토큰에 대한 예산* 을 더 사용했고요.

에이전트를 잘 만드는 데 *토큰을 아끼지 않는* 이유다 — 여기에 쓴 토큰이 *이후 모든 작업의 품질* 을 끌어올린다.

### 2단계 — 계획, 그리고 환각 줄이기

만든 에이전트에게 [`Plan mode`](/claude-code-클로드-코드-권한)로 *구글 로그인 추가 계획* 을 세우게 한다. 여기서 강사의 **프롬프트 팁** 이 빛난다.

> 궁금한 사항은 **질문해줘.**

왜 이게 중요할까? **AI 환각(Hallucination)** 때문이다. LLM은 *모르는 것도 아는 척* 지어낼 수 있다. 그래서 *"모르면 모른다고 말해줘"*, *"궁금하면 물어봐"* 라고 **여지를 열어주면
** — 환각이 줄어든다.

> 이건 마치 *시험에서 빈칸으로 두는 것* 보다 **찍어서라도 답을 내는 것** 과 마찬가지예요.

즉 AI는 *공백을 못 견디고* 뭐든 채우려 든다. **질문을 허용** 하는 것만으로, *억지 추측* 대신 **확인** 을 택하게 만드는 *영리한 안전장치* 다.

### 3단계 — 구현 (코드 + 콘솔 양쪽)

구글 로그인은 *코드만으로* 끝나지 않는다. **외부 설정** 이 함께 가야 한다.

```text
[코드 쪽]                          [설정 쪽]
OAuth 콜백 핸들러 작성        ┌─ Google Cloud Console
소셜 로그인 버튼 컴포넌트      │   프로젝트 생성 → OAuth 클라이언트 ID
로그인/회원가입 페이지 적용    └─ Supabase Authentication
                              Providers → Google 활성화 + Client ID·Secret
```

- **코드** → *OAuth 콜백 핸들러* + *소셜 로그인 버튼* → 로그인·회원가입 페이지에 적용 (에이전트가 처리)
- **Google Cloud Console** → *프로젝트 생성 → API 설정 → OAuth 클라이언트 ID* 발급
- **Supabase** → `Authentication → Providers` 에서 **Google 활성화** + *Client ID·Secret Key* 입력

*콘솔 설정* 은 사람이 해줘야 하는 영역이다 — [Supabase 세팅을 공식 문서로 했던 것](#supabase-프로젝트-생성하기)과 같은 맥락이다.

### 4단계 — 테스트 + DB 검증

구현했으면 *실제로 돌려본다.* 로그인·회원가입 페이지의 **구글 버튼** 으로 로그인 → **Protected 페이지로 리다이렉트** 되는지 확인한다.
그리고 [앞서 만든 트리거](#supabase-mcp-사용하기-로그인-구현하기) 덕에 — `profiles` 테이블·`Users` 에 **사용자 정보가 자동 등록** 됐는지 검증한다.

> 💡 **Provider 표시 차이.** *기존 이메일 가입 기록* 이 있는 계정과 *구글로만 로그인* 한 계정은, DB에 찍히는 **Provider 값** 이 다르다. 이런 디테일까지 확인하면 *인증 흐름* 이
> 더 또렷이 보인다.

### 정리하며 — 그리고 워크플로우 공식

구글 로그인 추가를 정리하면 다음과 같다.

- **① 에이전트** → `Next.js+Supabase 풀스택` → *최신 가이드·MCP* 참고 + [`ultrathink`](/claude-code-mvp-개발-및-배포) (첫 단추에 토큰 투자)
- **② 계획** → [`Plan mode`](/claude-code-클로드-코드-권한) + *"질문해줘"* → **환각 감소**
- **③ 구현** → *코드(콜백·버튼)* + *[Google Console](#supabase-프로젝트-생성하기) OAuth* + *Supabase Provider*
- **④ 검증** → 로그인 → 리다이렉트 → [`profiles` 자동 등록](#supabase-mcp-사용하기-로그인-구현하기) 확인 (+Provider 차이)

강사가 *시리즈 내내* 반복한 **개발 워크플로우 공식** 이 여기서 또 한 번 확인된다.

```text
정보 수집 → 서브 에이전트 생성 → 작업 계획 → 계획 기반 구현
```

기능이 *구글 로그인* 이든 무엇이든 — **이 순서는 변하지 않는다.** 토대를 깔고(이 시리즈), 전문가를 세우고, 계획하고, 구현한다. 앞으로는 이
위에서 [PRD·로드맵·Shrimp](/claude-code-mvp-개발-및-배포)로 *본격적인 기능* 들을 쌓아 올리면 된다.

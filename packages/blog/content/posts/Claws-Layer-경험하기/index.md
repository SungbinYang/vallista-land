## 🗂️ Claws 소개 (Open, Nano, Zero, Go 그리고 PaperClip)

이번 섹션에서는 AI 에이전트 시대에 새롭게 등장한 계층, **Claws** 에 대해 알아보려 한다. OpenClaw 같은 유명한 도구를 시작으로, 여기서 파생된 여러 프로젝트들을 하나씩 살펴볼 것이다.

참고로 이 분야는 변화가 정말 빠르다. 강사님의 표현을 빌리자면, *"OpenClaw 이후로 새로운 툴이 거의 한 시간에 하나 나온다고 해도 과하지 않을 정도"* 로 수많은 도구가 쏟아지고 있다.

### 그 전에: LLM, 에이전트, 그리고 Claws

Claws가 무엇인지 이해하려면 먼저 계층 구조를 짚고 가야 한다.

가장 아래에는 **LLM(거대 언어 모델)** 이 있다. LLM은 사실 그냥 *문자를 찍어내는 기계* 에 가깝다. 그래서 흔히 **"확률적 앵무새"** 라고 부르기도 한다.

그 위에 **AI 에이전트(Agent)** 라는 계층이 올라간다. 에이전트는 LLM에 **도구(Tool)** 를 쥐여주어, 단순히 글만 쓰는 것을 넘어 실제로 무언가를 *실행* 할 수 있게 만든 것이다. 이렇게
에이전트가 LLM을 통해 외부 도구를 불러 쓰는 것을 **Tool Calling** 이라고 한다. 그리고 여러 에이전트를 정리하고 호출하는 역할을 **오케스트레이션(Orchestration)** 이라고 부른다.

**Claws** 는 바로 이 에이전트 위에 한 겹 더 올라간 *새로운 계층* 이다. 스케줄링, 컨텍스트 관리, 툴 호출, 지속성(persistence) 등을 담당한다.

> 정리하면 **LLM → 에이전트 → Claws** 순으로 계층이 쌓여 올라간다고 이해하면 된다.

### OpenClaw — 유명하지만 보안이 걱정되는

**OpenClaw** 는 Claws의 가장 유명한 구현체 중 하나로, 과거에는 **Clawdbot** 이라 불렸다. OpenAI에 채용된 개발자가 만들었으며, **텔레그램(Telegram)** 을 통해 원격으로 내
컴퓨터에 지시를 내릴 수 있는 기능이 대표적이다.

아키텍처를 간단히 보면, 텔레그램 같은 커뮤니케이션 도구로 요청이 들어오면 → **Gateway** 를 거쳐 → **Agent** 를 실행하고 → Agent가 도구를 호출하며 메모리를 관리하는 구조이다.

다만 단점이 명확하다. **무려 40만 줄에 달하는 방대하고 엉성한 코드** 탓에 심각한 보안 취약점이 발견되었다. 강사님 역시 이 점을 강하게 지적한다.

> "40만 줄짜리 엉성한 코드로 괴물 같은 시스템이 되었는데, 거기에 내 데이터와 키를 맡기는 것은 당연히 좋지 않겠죠."

실제로 강사님은 OpenClaw를 설치했다가 바로 삭제한 경험도 공유한다.

> 한 가지 이름에 얽힌 이야기. Claws라는 이름은 원래 *Cloudbot* 에서 유래했는데, Anthropic의 **Claude** 와 상표권 문제가 생기면서 **OpenClaw** 로 바뀌었다. 'Claw'는
> *가재의 집게* 를 뜻하며, 그래서 많은 Claws 관련 프로젝트들이 가재 아이콘을 사용한다.

### NanoClaw — 가볍고 안전한 대안

그래서 강사님이 OpenClaw의 대안으로 추천하는 것이 **NanoClaw** 이다.

가장 큰 강점은 **경량성** 이다. 핵심 엔진이 단 **4,000줄** 정도의 코드로 구성되어 있어 관리하기 쉽다. 40만 줄의 OpenClaw와 비교하면 코드 양 자체가 비교가 안 될 만큼 적다.

코드가 적고 **의존성(dependency)이 낮다** 는 것은 보안 측면에서도 유리하다. 의존하는 라이브러리가 적을수록 **공급망 공격(Supply Chain Attack)** 에 노출될 위험이 줄어들기 때문이다.
(실제로 Axios 라이브러리에 악성 코드가 주입된 사건이 있었다.) 또한 NanoClaw는 별도의 설정 파일 없이 *스킬 단위* 로 관리된다는 특징이 있다.

### ZeroClaw — Rust로 만든 초경량 비서

**ZeroClaw** 는 **Rust** 언어로 만들어진 Claws 프로젝트로, *'Personal AI Assistant'* 를 표방한다.

가장 큰 특징은 압도적인 경량성이다. 무려 **10달러짜리 하드웨어, 5MB RAM** 에서도 실행이 가능하다. 그래서 라즈베리 파이 같은 소형 컴퓨터에서도 충분히 돌릴 수 있다.

> Rust는 *메모리 오너십(ownership) 시스템* 을 통해 메모리 노출과 관련된 보안 문제를 언어 차원에서 막아주는 것으로 잘 알려져 있다. 그래서 보안이 중요한 이런 프로젝트에 잘 어울린다.

### PicoClaw — 더 작은 기기까지

**PicoClaw** 역시 *Ultra-Lightweight AI* 를 강조하는 프로젝트로, ZeroClaw와 비슷하게 아주 적은 메모리만 차지한다. RISC-V 보드부터 안드로이드 핸드폰까지 다양한 기기에서
동작한다는 점이 특징이다.

### GoClaw — Go 언어 기반 플랫폼

**GoClaw** 는 이름 그대로 **Go 언어** 로 만들어진 AI 에이전트 플랫폼으로, 안전성과 확장성을 강조한다.

메모리 사용량이 적고 **Docker 이미지** 로도 활용할 수 있다. 또한 Anthropic, OpenAI와 호환되며 **프롬프트 캐싱(Prompt Caching)** 기능을 제공해 토큰 수를 절감할 수 있다는
장점이 있다.

### PaperClip — 에이전트로 '회사'를 운영하기

마지막은 조금 결이 다른 **PaperClip** 이다. 이는 오픈소스 **오케스트레이션** 도구이다.

PaperClip의 핵심 아이디어는 에이전트를 하나의 **'직원'** 으로 취급한다는 것이다. 각 에이전트에게 CEO, CPO, CTO 같은 페르소나를 부여하고, 마치 회사를 운영하듯 지시를 내릴 수 있다. 여러
에이전트를 팀으로 구성해 복잡한 작업을 분담시키는 데 유용하다.

강사님은 이 차이를 이렇게 정리한다.

> "OpenClaw가 하나의 직원이라면, PaperClip은 하나의 회사입니다."

### 마치며

이번 글에서는 LLM 에이전트 위에 올라간 새로운 계층 **Claws** 와, 그로부터 파생된 여러 프로젝트들을 살펴보았다. 간단히 정리하면 다음과 같다.

| 프로젝트          | 특징                               |
|---------------|----------------------------------|
| **OpenClaw**  | 텔레그램 연동, 유명하지만 40만 줄·보안 취약       |
| **NanoClaw**  | 4,000줄 경량, 낮은 의존성으로 안전 (추천)      |
| **ZeroClaw**  | Rust 기반, 10달러 하드웨어·5MB RAM에서도 구동 |
| **PicoClaw**  | 초경량, RISC-V부터 안드로이드까지            |
| **GoClaw**    | Go 기반, Docker 지원·프롬프트 캐싱         |
| **PaperClip** | 에이전트를 직원처럼 운영하는 오케스트레이션 도구       |

핵심은 **코드가 방대하고 의존성이 많을수록 보안 위험이 커진다** 는 점, 그래서 가볍고 의존성이 적은 프로젝트가 선호된다는 것이다. 다음 글에서는 추천 프로젝트인 **NanoClaw** 를 실제로 설치하고
그 구조를 들여다볼 예정이다.

## 🦞 2. 사용할 LLM API를 정하세요

본격적으로 Claws에 들어가기 전에, 먼저 짚고 넘어가야 할 것이 있다. 바로 **어떤 LLM API를 사용할지** 정하는 일이다. 에이전트를 오케스트레이션하려면 LLM API가 필수적이기 때문이다.

### 왜 이제 API 키가 중요해졌을까

예전에는 그냥 각 서비스에 로그인해서 쓰면 됐는데, 요즘은 사정이 좀 달라졌다. **LLM 밴더사(LLM을 만드는 회사)들이 점점 자사의 로그인 방식을 막고 있기 때문** 이다.

대표적으로 Anthropic의 경우, **OAuth 지원을 중단** 하고 추가 사용량에 대해서는 API로 결제하도록 정책을 바꿨다. 이런 흐름 때문에 이제는 API 키를 직접 발급받고 사용하는 방법을 이해하는 것이
점점 더 중요해지고 있다.

### OpenRouter — 하나의 키로 여러 모델을

첫 번째로 소개할 것은 **OpenRouter** 이다. 이름에서 짐작할 수 있듯, 마치 네트워크의 *라우터* 처럼 여러 LLM으로 요청을 분배해주는 역할을 한다. 즉, **API 키 하나만 발급받으면 여러 가지
LLM 모델을 모두 사용할 수 있는** 플랫폼이다.

OpenRouter의 강점은 단순히 여러 모델을 쓸 수 있다는 데 그치지 않는다. **Explorer Models** 메뉴에 들어가면 다양한 모델의 가격, 컨텍스트 길이(Context Length), 랭킹 등을
한눈에 비교할 수 있다. 무료이거나 무료에 가까운 LLM도 탐색할 수 있고, **리더보드** 를 통해 지금 가장 많이 쓰이고 가성비 좋은 모델이 무엇인지도 파악할 수 있다.

> 강사님은 "OpenRouter를 실제로 사용하지 않더라도, 현재 가성비가 가장 좋은 언어 모델이 무엇인지 바로 알 수 있어서 자주 들어와 확인하는 편"이라고 한다. 우리는 보통 코딩 플랜을 결제해두면 *그
> 모델이 최고* 라고 착각하기 쉬운데, 이렇게 객관적인 지표를 보면 시야가 넓어진다.

API 키를 발급할 때는 **Credit Limit(사용 한도)** 이나 **활성화 기간** 등을 함께 설정할 수 있어, 비용이 무한정 빠져나가는 것을 방지할 수 있다.

### Z.ai (GLM) — 가성비를 노린다면

두 번째는 **Z.ai** 이다. 이곳은 **GLM** 모델을 제공하는 회사인데, 가장 큰 매력은 **가성비** 이다.

Z.ai의 **Coding Plan** 을 이용하면, Claude Pro 요금제와 비교했을 때 훨씬 많은 토큰을 훨씬 저렴한 가격에 사용할 수 있다. 사실 GLM 모델은 앞서 소개한 OpenRouter에서도
사용할 수 있지만, 대량으로 쓸 거라면 Z.ai의 Coding Plan을 통해 직접 사용하는 편이 가성비 측면에서 더 유리할 수 있다.

> 가입 시 **Referral(추천) 코드** 를 활용하면 할인 혜택을 받을 수 있으니 참고하자.

### 빠르게 변하는 에이전트 생태계

LLM API뿐 아니라 에이전트 기술 자체도 정말 빠르게 변하고 있다. **Hermes Agent** 처럼 새로운 에이전트들이 끊임없이 등장하고 있는데, 그중에는 *사용량이 늘수록 점점 개선되는* 형태의
에이전트도 있다.

이렇게 새로운 도구가 나올 때 중요한 것은, 그 도구가 **어떤 모델을 지원하는지** 파악하는 것이다. 보통 **REST API 명세** 를 살펴보면 Opus, Gemini 등 어떤 모델을 지원하는지 확인할 수
있다.

### 마치며

이번 글에서는 Claws를 본격적으로 다루기 전에, 사용할 LLM API를 정하는 방법을 살펴보았다. 정리하면 다음과 같다.

- 밴더사들이 로그인 방식을 막으면서 **API 키 사용에 대한 이해** 가 중요해졌다.
- **OpenRouter** : 키 하나로 여러 모델을 쓰고, 가격·성능을 비교할 수 있다.
- **Z.ai (GLM)** : Coding Plan으로 가성비 높게 대량 사용이 가능하다.

가장 편하게 시작하는 방법은, **OpenRouter에 20~30달러 정도를 충전해두고 여러 모델을 직접 써보며 비교** 하는 것이다. 직접 경험해봐야 나에게 맞는 모델을 찾을 수 있기 때문이다.

## 🦞 OpenClaw + Telegram 설치와 사용

> ⚠️ 이 챕터는 강의 스크립트를 가져올 수 없어, 앞선 "Claws 소개" 챕터의 내용과 일반적인 텔레그램 봇 설정 절차를 바탕으로 재구성한 글이다. 실제 강의와 세부 명령어가 다를 수 있으니 참고용으로
> 봐주시길 바란다.

앞서 Claws를 소개하며 **OpenClaw** 를 잠깐 다뤘었다. 이번에는 그 OpenClaw를 직접 설치하고, **텔레그램(Telegram)** 과 연동해 사용하는 과정을 살펴보려 한다.

OpenClaw의 가장 큰 매력은 텔레그램으로 마치 *비서에게 메시지를 보내듯* 내 컴퓨터에 원격으로 지시를 내릴 수 있다는 점이다. 외출 중에 핸드폰으로 텔레그램 메시지를 보내면, 집에 있는 내 컴퓨터의
에이전트가 그 일을 대신 처리해주는 그림이다.

> ⚠️ **시작하기 전에 — 보안 주의**
> 앞 챕터에서도 강조했듯, OpenClaw는 40만 줄에 달하는 방대한 코드 탓에 보안 취약점이 지적되는 도구이다. 그래서 중요한 데이터나 핵심 API 키를 그대로 맡기는 것은 권장되지 않는다. 가능하면
> *테스트용 계정* 이나 *별도의 환경* 에서 경험해보는 것을 추천한다.

### 전체 흐름 미리 보기

설치와 연동은 크게 다음 순서로 진행된다.

1. **LLM API 키 준비** — 에이전트가 사용할 두뇌(모델)
2. **텔레그램 봇 생성** — 명령을 주고받을 창구
3. **OpenClaw 설치** — 본체
4. **환경 설정 연결** — API 키와 봇 토큰을 등록
5. **실행 및 테스트** — 텔레그램으로 말 걸어보기

하나씩 살펴보자.

### 1. LLM API 키 준비

OpenClaw도 결국 LLM을 두뇌로 사용하는 에이전트이다. 따라서 앞 챕터(*사용할 LLM API를 정하세요*)에서 발급받은 **API 키** 가 먼저 필요하다.

OpenRouter 키든, Z.ai(GLM)의 키든, Anthropic·OpenAI의 키든 자신이 사용할 모델의 키를 미리 준비해두자. 이 키는 잠시 뒤 환경 설정 단계에서 등록하게 된다.

### 2. 텔레그램 봇 만들기 (BotFather)

다음으로 명령을 주고받을 **텔레그램 봇** 을 만들어야 한다. 텔레그램에는 봇을 만들어주는 공식 봇인 **BotFather** 가 있다.

1. 텔레그램에서 `@BotFather` 를 검색해 대화를 시작한다.
2. `/start` 명령으로 BotFather가 제공하는 기능 목록을 먼저 확인한다.
3. `/newbot` 명령을 입력한다.
4. 안내에 따라 봇의 **이름(name)** 과 **사용자명(username)** 을 정한다. (사용자명은 보통 `_bot` 으로 끝나야 한다.)
5. 생성이 완료되면 BotFather가 **봇 토큰(Bot Token)** 을 발급해준다.

이때 한 가지 주의할 점이 있다. 봇 사용자명은 **전 세계에서 유일해야** 한다. 강사님도 이 부분을 짚는다.

> "텔레그램 봇은 전 세계에서 사용하다 보니, 쓸 수 있는 단어를 이미 다른 사람이 사용하고 있을 수도 있습니다."

그래서 원하는 이름이 이미 선점되어 있다면 여러 번 시도해야 할 수도 있다. 사용 가능한 이름을 찾으면, 그제서야 BotFather가 **HTTP API로 통신할 수 있는 토큰** 을 발급해준다. 이 봇 토큰은
에이전트가 텔레그램과 통신할 때 사용하는 열쇠이므로 잘 복사해두자.

> 🔑 봇 토큰은 비밀번호와 같다. 외부에 노출되면 누구나 내 봇을 조종할 수 있으니, 절대 깃허브 같은 공개된 곳에 올리지 않도록 주의하자.

### 3. OpenClaw 설치하기

이제 OpenClaw 본체를 설치한다. 보통 GitHub 저장소를 `git clone` 으로 받아오는 방식으로 설치한다.

```bash
# OpenClaw 저장소를 복제
git clone <OpenClaw 저장소 주소>
cd openclaw

# 의존성 설치 (Node.js 기반인 경우)
npm install
```

> 앞서 배운 `git clone` 이 여기서도 쓰인다. 원격 저장소를 통째로 내 컴퓨터로 복제해오는 것이다.

### 4. 환경 설정 연결하기

설치가 끝나면, 준비해둔 **LLM API 키** 와 **텔레그램 봇 토큰** 을 OpenClaw에 등록해야 한다. 보통 프로젝트 루트에 `.env` 같은 환경 변수 파일을 만들어 값을 넣어준다.

```bash
# .env 예시
LLM_API_KEY=발급받은_LLM_API_키
TELEGRAM_BOT_TOKEN=BotFather가_준_봇_토큰
```

> `.env` 파일에는 민감한 키들이 담기므로, `.gitignore` 에 반드시 추가해 깃에 올라가지 않도록 하자. (Git 챕터에서 배운 그 `.gitignore` 가 여기서 중요한 역할을 한다.)

### 5. 실행하고 텔레그램으로 말 걸어보기

이제 OpenClaw를 실행한다.

```bash
npm start
```

정상적으로 실행되면, 아까 만든 텔레그램 봇과의 대화창으로 들어가 메시지를 보내보자. 예를 들어 "오늘 날씨 정리해줘" 같은 간단한 지시부터 시작해볼 수 있다.

그러면 앞 챕터에서 본 구조대로 동작한다.

> **텔레그램(메시지)** → **Gateway** → **Agent 실행** → **Tool Calling(도구 호출)** → **Memory 관리** → **응답 반환**

내가 보낸 메시지가 게이트웨이를 거쳐 에이전트를 깨우고, 에이전트가 필요한 도구를 호출해 작업을 수행한 뒤, 그 결과를 다시 텔레그램으로 돌려주는 것이다.

### 마치며

이번 글에서는 OpenClaw를 설치하고 텔레그램과 연동해 사용하는 과정을 살펴보았다. 정리하면 다음과 같다.

- **LLM API 키 → 텔레그램 봇 토큰 → OpenClaw 설치 → 환경 설정 → 실행** 의 순서로 진행한다.
- 텔레그램 봇은 **BotFather** 의 `/newbot` 으로 만들고, 발급된 **봇 토큰** 을 OpenClaw에 등록한다.
- API 키와 봇 토큰 같은 민감 정보는 `.env` 에 넣고 `.gitignore` 로 보호한다.

다시 한번 강조하지만, OpenClaw는 강력하고 재미있는 도구이지만 보안 측면에서는 신중해야 한다. 그래서 강사님도 더 가볍고 안전한 **NanoClaw** 를 추천했던 것이다. OpenClaw로 *원격 에이전트가
어떤 느낌인지* 경험해본 뒤에는, 더 안전한 대안으로 옮겨가는 것을 고려해보자.

## 👼 Hermes Agent 설치와 연결

> 강의가 개편되면서, 텔레그램 원격 에이전트 실습이 OpenClaw에서 **Hermes Agent** 로 옮겨졌다. (강의 제목도 *"…Hermes Agent 포함…"* 으로 바뀌었다.) 이번 섹션은 실제
> [Hermes Agent 설치와 연결](https://www.inflearn.com/courses/lecture?courseId=341165&unitId=464851) 강의 내용을 바탕으로 정리한 글이다.

앞서 OpenClaw로 *텔레그램을 통해 내 컴퓨터에 원격으로 지시하는* 그림을 그려봤다. **Hermes Agent** 는 바로 그 경험을, 더 정돈된 형태로 제공하는 에이전트이다. 강사님이 실습에서 **맥미니(
Mac
mini)** 를 대상으로 사용하는 도구가 바로 이것이다.

Hermes Agent는 스스로를 이렇게 소개한다.

> **"The Agent That Grows With You"** — 당신과 함께 성장하는 에이전트.

즉, 텔레그램 같은 인터페이스로 명령을 내리면, 맥미니에 있는 에이전트가 그 일을 실시간으로 처리해주는 구조이다. 앞서 본 원격 에이전트의 개념과 완전히 같다.

### 설치 방법은 두 가지

Hermes Agent는 두 가지 방식으로 설치할 수 있다. 자신에게 편한 쪽을 고르면 된다.

1. **터미널(Homebrew)** — 명령어로 설치하고 설정하는 방식
2. **GUI 앱** — 맥용 앱을 내려받아 시각적으로 설치·설정하는 방식

강사님은 *"기본적으로 Homebrew를 좋아하기 때문에"* 터미널 방식을 먼저 보여준 뒤, GUI 방식도 함께 소개한다. 하나씩 살펴보자.

### 방법 1. 터미널(Homebrew)로 설치하기

먼저 공식 홈페이지에 접속해 설치 안내를 확인한 뒤, Homebrew로 설치한다. (Homebrew는 앞선 챕터에서 배운 그 macOS 패키지 매니저이다. 업데이트 관리가 쉬워 강사님이 선호한다.)

```bash
# Homebrew가 설치되어 있는지 먼저 확인
brew --version

# Hermes Agent 설치
brew install hermes

# 작업용 디렉토리를 만들고 이동
mkdir hermes && cd hermes

# 실행 → 초기 설정 시작
hermes
```

`hermes` 를 실행하면 대화형으로 **초기 설정** 이 진행된다. 순서대로 다음 값들을 입력하게 된다.

- **GLM API 키** — Zhipu AI(GLM)의 모델을 사용하기 위한 인증 키. 앞 챕터에서 준비한 코딩 플랜 키를 넣으면 된다. (모델은 **GLM-Turbo** 사용을 권장한다.)
- **로컬 환경 설정** — 맥미니 같은 로컬 환경에 대한 기본 설정
- **Bot Token** — 앞에서 BotFather로 발급받은 텔레그램 봇 토큰
- **Allowed Telegram User ID** — 이 에이전트에 명령을 내릴 수 있는 **허용된 텔레그램 사용자 ID**
- **검색 엔진** — Brave Search, DuckDuckGo 등 에이전트가 사용할 검색 엔진 선택

> ⚠️ **가장 흔한 실수 — User ID는 반드시 '숫자'**
> `Allowed Telegram User ID` 에는 사용자명(@아이디)이 아니라 **숫자로 된 ID** 를 넣어야 한다. 강사님도 여기서 형식을 잘못 입력해 오류를 겪는 장면을 그대로 보여준다. 숫자가 아닌
> 값을 넣으면
> 연동이 제대로 되지 않으니 주의하자.

설정을 모두 마치면 Hermes Agent가 실행된다.

### 방법 2. GUI 앱으로 설치하기

터미널이 익숙하지 않다면, **맥용 앱** 을 내려받아 GUI 환경에서 설치할 수도 있다.

1. 맥용 Hermes Agent 앱을 다운로드해 실행한다.
2. **`Install Hermes`** 를 선택하면 시스템 설치가 진행된다.
3. **`Messaging`** 섹션으로 들어가 텔레그램 설정을 확인한다.
4. 여기서도 **Bot Token** 과 **Allowed Telegram User ID** 를 입력한다. (역시 User ID는 **숫자** 여야 한다.)

터미널이든 GUI든 입력하는 핵심 값(GLM 키, Bot Token, 숫자 User ID)은 동일하다는 점만 기억하면 된다.

### 오류가 나면? 에이전트에게 직접 물어보기

설정 과정에서 오류가 나더라도 당황할 필요는 없다. 강사님이 보여주는 재미있는 해결법이 있다. 바로 **모델(GLM-Turbo)을 연결한 뒤, 에이전트에게 직접 무엇이 문제인지 물어보는 것** 이다. 에이전트가
스스로 오류의 원인과 해결책을 알려주기도 한다. AI 에이전트다운 문제 해결 방식이다.

또 하나, 텔레그램 메시지가 **실시간으로 반영되지 않을 때** 는 앱을 껐다가 다시 켜주면 대체로 해결된다.

### 실제로 말 걸어보기

설정이 끝났다면, 텔레그램 봇 대화창에서 실제로 지시를 내려볼 차례이다. 강사님은 예시로 **"하이닉스 종가 알려줘"** 같은 요청을 던진다. 그러면 에이전트가 검색 도구를 호출해 실제 주가 정보를 찾아
답해준다.

> 앞서 배운 **Tool Calling(도구 호출)** 이 바로 이 장면에서 동작한다. 단순히 글만 쓰는 것이 아니라, 필요한 도구를 직접 불러 *실제 작업* 을 수행하는 것이다.

### 마치며

이번 글에서는 텔레그램 원격 에이전트 실습을 **Hermes Agent** 로 직접 경험해보았다. 정리하면 다음과 같다.

- Hermes Agent는 **텔레그램으로 맥미니에 원격 명령** 을 내릴 수 있는 AI 에이전트이며, *"함께 성장하는 에이전트"* 를 표방한다.
- 설치는 **터미널(`brew install hermes`)** 과 **GUI 앱** 두 가지 방식이 있고, 입력하는 핵심 값(**GLM 키 · Bot Token · 숫자 User ID**)은 동일하다.
- `Allowed Telegram User ID` 는 반드시 **숫자** 로 넣어야 하며, 오류가 나면 **에이전트에게 직접 물어** 해결할 수 있다.
- 설정을 마치면 텔레그램으로 *"하이닉스 종가 알려줘"* 처럼 실제 작업을 지시할 수 있다.

여기까지 따라왔다면, 이제 손 안의 텔레그램만으로 내 컴퓨터의 에이전트에게 일을 시키는 경험을 직접 해본 셈이다. 처음 소개했던 **LLM → 에이전트 → Claws** 계층이, 이렇게 실제 도구로 손에 잡히게
된다.

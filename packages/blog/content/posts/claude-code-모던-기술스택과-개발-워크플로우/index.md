## 🧱 모던 기술 스택 - 기초

여기까지 우리는 클로드 코드의 *스펙* 을 하나씩 훑어왔다. [슬래시 커맨드](/claude-code-슬래시-명령어와-단축키), [권한](/claude-code-클로드-코드-권한),
[설정·메모리](/claude-code-설정-파일과-메모리-관리), [MCP](/claude-code-mcp-활용)까지. 그런데 한 가지 짚고 넘어가자.

> 그러면 이제 Claude Code를 모두 배웠냐? **절대 아닙니다.**

스펙을 *아는 것* 과, 그걸 *실제 개발에 녹여 쓰는 것* 은 다른 이야기다. 커스텀 슬래시 커맨드·서브 에이전트·훅 같은 *고급 기능* 도 아직 남아 있다. 이번 섹션부터는 그 고급 기능들을 **하나의 프로젝트
** 에
*녹여내며* 배운다. 개별 스펙을 넘어 **개발 워크플로우** 자체를 익히는 게 목표다.

### 무엇을 만드나 — 웹 스타터 킷

이번 섹션부터 함께 만들 결과물은 **Claude Code + Next.js 웹 스타터 킷** 이다.

> 스타터 킷은 개발 시작에 필요한 *기본 파일과 설정* 을 미리 구성해둔 소스 코드예요.

쉽게 말해 *매번 새 프로젝트마다 반복* 하는 초기 세팅(폴더 구조, 설정, 공통 컴포넌트…)을 **한 번 잘 만들어두고 재사용** 하는 *틀* 이다. *왜* 지금 이게 중요할까?

> 지금은 어떤가요? AI의 도움으로 개발자 한 명이 *여러 개의 프로젝트를 동시에* 진행할 수 있는 세상이 되었습니다.

AI와 함께라면 *혼자서도* 여러 서비스를 굴릴 수 있는 시대다. 이때 **잘 만든 스타터 킷 하나** 는 *프로젝트를 새로 깔 때마다* 드는 시간을 *통째로* 아껴준다. 그 첫걸음이 바로 **기술 스택 선택**
이다.

### 복잡도로 나누는 기술 스택

기술 스택에 *정답* 은 없다. **만들 것의 복잡도** 에 맞춰 고르는 게 핵심이다. 강의에서는 복잡도를 *네 단계* 로 나눠 제시한다. *웹 애플리케이션* 은 다시 **로그인 유무** 와 **데이터베이스 사용
여부** 로 초급·중급·고급이 갈린다.

| 복잡도            | 예시             | 핵심 스택                              |
|----------------|----------------|------------------------------------|
| **간단한 홈페이지**   | 소개 페이지, 랜딩 페이지 | HTML · JS · CSS + **Tailwind CSS** |
| **초급** 웹앱 (읽기) | 개인 블로그, 뉴스 사이트 | **Next.js** + CMS(Notion API 등)    |
| **중급** 웹앱      | 로그인 O          | + 인증 / 데이터베이스 *(심화 챕터)*            |
| **고급** 웹앱      | 로그인 O + 복잡한 DB | + 본격 백엔드 *(심화 챕터)*                 |

이번 *기초* 챕터에서는 **간단한 홈페이지** 와 **초급(읽기 전용) 웹앱** 두 가지를 다룬다. 중급·고급은 다음 *심화* 챕터로 넘긴다.

### ① 간단한 홈페이지 — Tailwind CSS

*소개 페이지* 처럼 **정적인 한 장짜리** 페이지라면, 굳이 무거운 프레임워크가 필요 없다. **HTML · JavaScript · CSS** 기본기에 **Tailwind CSS** 만 얹으면 충분하다.

> Next.js 같은 프레임워크는 *간단한 홈페이지* 엔 **오버 스펙** 이 될 수 있어요.

여기서 **Tailwind CSS** 는 *유틸리티 클래스* 기반의 스타일링 도구다. `flex`, `pt-4`, `text-center` 처럼 *작은 클래스들을 조합* 해 *별도 CSS 파일 없이* UI를 만든다.

```html

<button class="px-4 py-2 rounded-lg bg-blue-600 text-white hover:bg-blue-700">
    시작하기
</button>
```

이런 *클래스 조합* 만으로 *멋진 버튼* 이 뚝딱 나온다. 무엇보다 **생성형 AI와 궁합이 좋다.**

> 다양한 생성형 AI에서도, 별도 지침이 없으면 *Tailwind 기반* 으로 UI를 만들어주는 경우가 많아요.

즉 클로드 코드에게 *"버튼 만들어줘"* 하면 *대개 Tailwind로* 짜준다는 뜻이다. AI 시대의 *사실상 표준* 스타일링이라 익혀두면 손해 볼 일이 없다.

### ② Read 전용 웹앱 — Next.js + CMS

한 단계 올라가, **개인 블로그** 나 **뉴스 사이트** 처럼 *콘텐츠를 동적으로 보여주는* 웹앱을 보자. 이게 *초급* 웹앱, 즉 **읽기(Read) 전용** 애플리케이션이다.

여기서 *오해* 하기 쉬운 지점이 있다. *블로그* 라고 해서 *복잡한 데이터베이스* 부터 떠올릴 필요가 없다는 것.

> 복잡한 데이터베이스 대신, **Notion API** 같은 *CMS* 를 활용하면 돼요.

**CMS(콘텐츠 관리 시스템)** 는 *글·이미지 같은 콘텐츠* 를 관리해주는 도구다. *Notion에 글을 쓰면*, 그걸 **Notion API** 로 불러와 화면에 *동적으로* 뿌리는 식이다. DB 서버를
*직접 운영하지
않아도* 된다.

그리고 이 구조의 **핵심** 이 바로 **Next.js** 다.

| Next.js의 장점   | 의미                           |
|---------------|------------------------------|
| **파일 기반 라우팅** | *폴더·파일 구조* 만으로 URL 경로가 잡힘    |
| **성능 최적화**    | 렌더링·이미지 등 *모던 웹에 필요한 최적화* 내장 |
| **활발한 생태계**   | 커뮤니티·라이브러리·*채용 시장* 까지 유리     |

특히 *생태계* 는 무시 못 할 강점이다. **shadcn/ui**, **Supabase** 같은 *주요 도구* 들도 *Next.js 기반 개발을 권장* 한다.

> 별도 지침이 없다면, 생성형 AI도 웹 애플리케이션은 *Next.js 기반* 으로 가이드해줄 거예요.

*많이 쓰이는 만큼* AI도 *잘 알고*, *참고 자료* 도 풍부하다 — 그래서 **AI와 함께 개발하기에 최적** 인 프레임워크인 셈이다.

### 스타일링 한 스푼 — shadcn/ui

Tailwind만으로도 충분하지만, *버튼·카드·모달* 처럼 *자주 쓰는 컴포넌트* 를 매번 클래스로 조립하긴 번거롭다. 그래서 더하는 게 **shadcn/ui** 다.

**shadcn/ui** 는 *Tailwind 기반* 의 **컴포넌트 라이브러리** 다. *자주 쓰는 UI 컴포넌트* 를 *미리 예쁘게* 정의해두어, **Tailwind의 장점은 그대로 누리면서** *완성도 높은
UI* 를
*빠르게* 얹을 수 있다.

정리하면 **Tailwind**(유틸리티 스타일링) 위에 **shadcn/ui**(완성형 컴포넌트)를 얹는 — 이 *조합* 이 *읽기 전용 웹앱* 의 **표준 스타일링 스택** 이다.

### 정리하며

모던 기술 스택 *기초* 를 정리하면 다음과 같다.

- **이번 섹션의 목표** → 스펙을 *아는 것* 을 넘어, **개발 워크플로우** 에 녹여 쓰기 (결과물: **Next.js 웹 스타터 킷**)
- **기술 스택은 복잡도로** → *간단한 홈페이지* < *읽기 전용* < *중급(로그인)* < *고급(DB)*
- **간단한 홈페이지** → HTML·JS·CSS + **Tailwind CSS** (Next.js는 *오버 스펙*)
- **Read 전용 웹앱** → **Next.js**(파일 라우팅·성능·생태계) + **CMS**(Notion API 등)
- **스타일링** → **Tailwind** + **shadcn/ui**(완성형 컴포넌트)

기술 스택의 *큰 그림* 을 그렸으니, 다음 **심화** 챕터에서는 *중급·고급* 웹앱 — 즉 *로그인* 과 *데이터베이스* 가 들어가는 스택까지 이어서 살펴보자.

## 🏗️ 모던 기술 스택 - 심화

[기초 챕터](#-모던-기술-스택---기초)에서 *간단한 홈페이지* 와 *읽기 전용 웹앱* 까지 봤다. 이제 *복잡도* 를 한 단계씩 더 올려, **중급(로그인·폼)** 과 **고급(데이터베이스)** 웹앱의 스택을
채워보자.

> 리드(Read) 전용 스택에서는 *읽기만* 했다면, **폼 기능** 을 활용해서 *등록·수정* 처럼 **쓰기도** 가능한 거죠.

핵심은 *읽기 전용 스택을 버리는 게 아니라*, 그 위에 **필요한 조각만 얹는다** 는 것이다. [Next.js·Tailwind·shadcn/ui](#-read-전용-웹앱--nextjs--cms) 토대는
그대로 두고, *인증* 과 *데이터베이스* 를 더한다.

### 중급 웹앱 — 인증 서비스

**중급** 은 *로그인이 필요한* 웹앱이다. *회원가입·로그인* 으로 **페이지 접근을 제어** 하고, *폼* 으로 데이터를 등록·수정한다. 여기서 새로 더해지는 게 **인증(Authentication) 서비스**
다.

인증은 *직접 구현하면* 보안 이슈가 많아 *까다롭다.* 그래서 **검증된 서비스** 를 가져다 쓰는 게 정석이다. 상황에 따라 *네 가지* 중 고르면 된다.

| 인증 서비스            | 언제 고르나                           |
|-------------------|----------------------------------|
| **Clerk**         | *개발 속도* 와 **UI 완성도** 가 최우선일 때    |
| **Firebase Auth** | **Google 생태계** 활용 + *빠른 개발*      |
| **NextAuth.js**   | **Next.js** 안에서 *간편하게* 구현하고 싶을 때 |
| **Supabase Auth** | *백엔드 확장* 까지 염두에 둘 때 (Supabase로)  |

*정답* 은 없다. **무엇을 중시하느냐** — 완성도냐, 구글 연동이냐, Next.js 통합이냐, 백엔드 확장이냐 — 에 따라 고르면 된다.

### 폼 관리 — React Hook Form + Zod

*회원가입* 이든 *글쓰기* 든, 중급 웹앱엔 **폼** 이 빠지지 않는다. 폼을 *맨손으로* 짜면 *상태 관리·유효성 검사* 코드가 *주렁주렁* 길어진다. 그래서 강의가 추천하는 **조합** 이 있다.

| 라이브러리               | 역할                               |
|---------------------|----------------------------------|
| **React Hook Form** | 폼 *상태 관리* — **코드량을 획기적으로** 줄인다   |
| **Zod**             | *스키마 기반* **유효성 검사** (TypeScript) |

이 둘을 *함께* 쓰면 *적은 코드* 로 *안전한 폼* 이 나온다. 그런데 여기엔 **AI 개발만의 숨은 이점** 이 하나 더 있다.

> 코드량이 줄어든다는 건, **Claude Code 같은 AI 도구를 쓸 때도** 그만큼 *컨텍스트가 줄어든다* 는 거겠죠?

코드가 *짧을수록* 클로드 코드가 *읽고 다뤄야 할* [컨텍스트](/claude-code-슬래시-명령어와-단축키)도 *작아진다.* 즉 이 조합은 *애플리케이션을 안전하게* 만들 뿐 아니라, **AI의 컨텍스트
효율** 까지 끌어올린다. *바이브 코딩* 에 *유리한* 선택인 셈이다.

### 고급 웹앱 — Supabase

마지막 **고급** 은 *본격적인 데이터베이스* 가 들어가는 웹앱이다. *왜* DB가 필요할까?

> 테트리스 게임에서 *내 점수* 를 *다른 사람과 공유* 하려면, 그 점수를 *어딘가 저장* 해둬야 하잖아요.

*사용자끼리 데이터를 공유·축적* 하려면 **저장소(DB)** 가 있어야 한다. 이때 강의가 미는 건 **Supabase** 다.

**Supabase** 는 **완전 관리형 백엔드(BaaS)** 다. *DB 서버를 직접 깔고 운영* 하는 대신, *필요한 백엔드 기능* 을 *한 곳에서* 제공한다.

- **데이터베이스** — PostgreSQL 기반
- **인증(Auth)** — 앞서 본 *Supabase Auth* 가 여기에 포함
- **스토리지** — 이미지·파일 저장
- **서버리스 함수** · **실시간(Realtime)** 기능

*인증부터 DB·스토리지까지* **하나로** 묶이니, *읽기 전용 → 중급 → 고급* 으로 올라가도 **같은 생태계** 안에서 확장된다. 그래서 *고급 스택* 의 중심으로 자주 추천된다.

### 기술 스택 선택 전략 — 그리고 AI 활용

스택을 *네 단계* 로 훑었다. 그런데 강의가 *진짜로* 강조하는 건 *그다음* 이다. **개발 경험 유무** 에 따라 이 내용을 *받아들이는 법* 이 갈린다는 것.

> 개발 경험이 있으신 분들은 물음표가 아니라, *"맞아, 이렇게 개발하면 되지"* 하고 느끼셨을 거예요.

| 상황                      | 전략                                      |
|-------------------------|-----------------------------------------|
| **개발 경험 O**             | 익숙한 스택이면 *그대로* — 더 나은 조합으로 *직접 설계* 도 가능 |
| **개발 경험 X** (HTML·CSS…) | 막막해도 괜찮다 → **Claude Code에게 *맥락* 을 제공**  |

> 여기서 HTML·CSS·Tailwind가 *무엇인지 모르신다면*, 이건 *어쩔 수 없습니다.* 대신 **Claude Code에게 맥락을 제공** 하는 방식으로 활용하면 돼요.

즉 *기초가 부족해도* — [`CLAUDE.md`·메모리](/claude-code-설정-파일과-메모리-관리)나 [MCP](/claude-code-mcp-활용)로 *맥락을 채워주면* — 클로드 코드와 함께
개발할 수 있다는 뜻이다.

그리고 *마지막 당부.* 여기서 소개한 스택은 **범용적이고 인기 있는** 기준일 뿐, **정답이 아니다.**

> 팀이나 회사 상황에 따라 *프론트와 백엔드를 분리* 하거나, *Java 기반 Spring* 같은 다른 스택을 골라도 됩니다.

*복잡한 요구사항* 이라면 **스스로 스택을 설계** 할 수도 있어야 한다. 강의의 선택은 *출발점* 일 뿐, *내 상황에 맞게* 갈아끼우는 *유연함* 이 핵심이다.

### 정리하며

모던 기술 스택 *심화* 를 정리하면 다음과 같다.

- **중급(로그인·폼)** → 읽기 전용 스택 + **인증 서비스**
    - **Clerk**(UI 완성도) · **Firebase Auth**(구글 생태계) · **NextAuth.js**(Next.js 통합) · **Supabase Auth**(백엔드 확장)
- **폼** → **React Hook Form + Zod** → *코드량↓ → 컨텍스트↓ →* **AI 효율↑** + 안전성↑
- **고급(DB)** → **Supabase**(완전 관리형 백엔드: DB·인증·스토리지·실시간)
- **선택 전략** → *경험 O* 면 직접 설계 / *경험 X* 면 **클로드에게 맥락 제공**
- 소개된 스택은 **정답이 아닌 출발점** → 상황 따라 *Spring* 등으로 *유연하게*

이로써 **모던 기술 스택** 의 *큰 그림* 을 모두 그렸다. *무엇을* 만들지 정했으니, 다음 챕터에서는 이 스택 위에서 클로드 코드와 *어떻게* 일할지 — **`CLAUDE.md` 설정과 AI 개발 워크플로우
**
— 로 넘어가 보자.

## 🧭 CLAUDE.md 설정 및 AI 개발 워크플로우

기술 스택으로 *무엇을* 만들지 정했다면, 이제 클로드 코드와 *어떻게* 일할지를 잡을 차례다. 그 출발점은 **유저 메모리 설정** 이고, 본론은 **Anthropic이 권장하는 AI 개발 워크플로우** 다.

### 유저 메모리부터 — 한국어로 대화하기

[메모리 챕터](/claude-code-설정-파일과-메모리-관리)에서 봤듯, **유저 메모리** 는 *내 모든 프로젝트* 에 공통 적용되는 컨텍스트다. 위치는 **`~/.claude/CLAUDE.md`** 다.
*어느
프로젝트* 에서나 지켜야 할 규칙을 여기 적어둔다.

가장 먼저 적어둘 만한 건 *"한국어로 대화하기"* 다.

```markdown
# ~/.claude/CLAUDE.md

- 항상 **한국어로** 답변할 것
```

> 코딩 경험이 없으신 분들은, 이러한 **터미널** 이 나왔다고 해서 *두려워하시면 안 됩니다.*

설정은 *터미널* 로 `~/.claude` 폴더의 `CLAUDE.md` 를 [Cursor](/claude-code-cursor-ai-ide-통합) 같은 편집기로 열어 적으면 된다. *한 줄짜리 설정* 이지만,
이걸 *유저 메모리* 에 둔 덕분에 *새 프로젝트마다 다시* 부탁할 필요가 없어진다.

### 왜 '워크플로우'가 필요한가 — 건축 비유

설정을 마쳤으니, *본격적인 질문* 으로 가자. **왜 그냥 시키면 안 되고, '워크플로우'가 필요할까?**

> 건축가에게 그냥 *"좋은 건물 지어주세요"* 라고 하면 어떨까요? 원하는 결과가 나오기 어렵겠죠.

*"멋진 건물"* 이 아니라 — *몇 층인지, 용도는 무엇인지, 예산은 얼마인지* — **구체적인 요구사항** 을 줘야 원하는 결과가 나온다. 소프트웨어도 똑같다. *전문 지식* 과 *도메인 지식* 이 *맥락
(컨텍스트)* 으로 들어가야 클로드가 *제대로* 만든다.

> 개발 지식이 부족하더라도, **AI 개발 워크플로우를 잘 활용** 하면 원하는 결과물을 얻을 확률이 *훨씬 높아집니다.*

즉 *워크플로우* 는 *전문 지식의 빈자리* 를 메우는 **틀** 이다. 이 틀을 따르면 *지식이 부족해도* 성공 확률이 올라간다.

### Anthropic 공식 4단계 — 탐색·계획·구현·커밋

강의가 소개하는 건 **Anthropic 공식 문서가 권장** 하는 *4단계* 워크플로우다. *바로 코딩부터* 시작하지 않는다는 게 핵심이다.

| 단계                | 하는 일                                        |
|-------------------|---------------------------------------------|
| **① 탐색**(Explore) | 관련 *파일·이미지·URL* 을 읽혀 **도메인 지식(컨텍스트)** 을 쌓는다 |
| **② 계획**(Plan)    | 복잡한 문제를 *단계별로* 쪼개 **계획** 을 세운다              |
| **③ 구현**(Code)    | 계획대로 *코드를 짜고*, 각 단계를 **검증** 한다              |
| **④ 커밋**(Commit)  | 완성된 코드를 *안전하게* **저장(커밋)** 한다                |

각 단계를 조금 더 풀어보자.

- **① 탐색** — *코딩에 앞서* 클로드에게 *관련 파일·이미지·URL* 을 읽게 한다. 이렇게 쌓은 **컨텍스트** 가 결과물의 질을 좌우한다.
- **② 계획** — [플랜 모드](/claude-code-클로드-코드-권한)와 **`Think`** 키워드를 활용해, *복잡한 작업* 을 *단계별로* 설계한다. *깊이 생각* 하게 만드는 단계다.
- **③ 구현** — 계획에 따라 *실제 코드* 를 짠다. *한 번에 몰아치기* 보다, *각 구현을 검증하며* 진행하는 게 좋다. *타입 체크* 나 [MCP](/claude-code-mcp-활용) 활용이 *검증
  수단* 이 된다.
- **④ 커밋** — 완성된 코드를 [Git으로 커밋](/claude-code-git과-github)해 *안전하게 보관* 한다.

### ⭐ 핵심은 '탐색'과 '계획'

이 워크플로우에서 *가장 중요한* 두 단계는 **탐색** 과 **계획** 이다.

> 그렇기 때문에 이번 강의에서는 *여러 번 강조* 드릴 거예요.

*조급한 마음* 에 **③ 구현부터** 뛰어들고 싶겠지만, *복잡한 작업일수록* **앞의 두 단계** 를 충실히 거쳐야 성공 확률이 높다. *맥락 없이* 짜낸 코드는 *엉뚱한 방향* 으로 가기 쉽다. **충분히
탐색하고, 차분히 계획한 뒤, 구현으로** — 이 순서를 *몸에 익히는 게* 이 챕터의 전부라 해도 좋다.

그리고 이 흐름은 *내가 임의로* 정한 게 아니라 **Anthropic 공식 가이드** 를 따른 것이니, *믿고* 익혀도 된다.

### 정리하며

`CLAUDE.md` 설정과 AI 개발 워크플로우를 정리하면 다음과 같다.

- **유저 메모리**(`~/.claude/CLAUDE.md`) → *모든 프로젝트 공통* 컨텍스트, 예: **한국어로 답변**
- **왜 워크플로우?** → *"좋은 건물 지어줘"* 로는 부족 → **구체적 맥락** 을 줘야 원하는 결과
- **Anthropic 4단계** → **탐색 → 계획 → 구현 → 커밋**
- ⭐ **핵심은 탐색·계획** → *바로 코딩 X*, 복잡할수록 앞 두 단계가 성공을 가른다

이로써 **모던 기술 스택과 개발 워크플로우** 섹션을 마친다. *무엇을*(기술 스택) 만들지, *어떻게*(4단계 워크플로우) 일할지 — 두 축을 손에 넣었다. 다음 섹션부터는 이 워크플로우를 *실제 프로젝트* 에
얹어, **모던하고 예쁜 UI** 까지 만들어가는 여정을 이어가 보자.

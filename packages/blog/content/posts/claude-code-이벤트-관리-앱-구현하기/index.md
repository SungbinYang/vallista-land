## 작업 구현: 애플리케이션 골격 구축

기획(PRD·ROADMAP·린캔버스)을 마치고, Shrimp Task Manager로 작업까지 잘게 쪼개 두었습니다. 이제는 그 계획을 **실제 코드로 옮길** 차례입니다. 이번 챕터에서는 1단계 작업인 *
*애플리케이션 골격 구축(Phase 1)** 을 Claude Code와 함께 실행해 봅니다. 페이지를 빈 껍데기로 만들고, 레이아웃을 다듬고, 타입을 정의한 뒤, 코드 리뷰와 커밋까지 마무리하는 한 사이클을 통째로
경험하는 시간입니다.

> "우리가 이전 시간에 세운 작업 계획을 구현해 보도록 하겠습니다."

### 골격 구축이란 무엇인가

본격적으로 기능을 붙이기 전에, 먼저 **애플리케이션의 전체 윤곽부터** 잡아 둡니다. 전체 레이아웃과 관련된 컴포넌트를 만들고, 각 페이지를 **내용이 비어 있는 껍데기 상태로** 생성해 두는 작업입니다.

> "즉, 애플리케이션에 대한 전체 골격을 구축하는 작업이라고 보시면 됩니다."

이렇게 골격을 먼저 세워 두면, 화면 간 이동 흐름과 전체 구조를 눈으로 확인한 뒤에 살을 붙여 나갈 수 있습니다. 앞 챕터에서 1단계 작업을 **9개의 서브 태스크로 분할**해 두었으니, 이제 그 작업들을 하나씩
실행하기만 하면 됩니다.

### `execute-task`로 페이지 생성하기

분할해 둔 작업을 실행하는 명령이 바로 `execute-task` 입니다. 강의에서는 페이지 생성과 관련된 **1번부터 4번 작업까지**를 한 번에 구현하도록 요청합니다. 반복적인 확인 절차로 흐름이 끊기지 않도록
`Bypass permissions` 모드로 진행합니다.

> "그러면 이러한 작업을 `execute-task`로 진행하게 될 텐데요."

작업이 끝나면 모바일 사용자 페이지부터 관리자 페이지까지 **총 18개의 페이지가 생성**된 것을 확인할 수 있습니다. 막연했던 구조가 실제 파일과 라우트로 모습을 드러내는 순간입니다.

| 단계     | 내용                               |
|--------|----------------------------------|
| **명령** | `execute-task` 로 1~4번 작업 실행      |
| **모드** | `Bypass permissions` 로 확인 절차 최소화 |
| **결과** | 사용자·관리자 영역을 아우르는 18개 페이지 생성      |

### 개발 서버로 바로 확인하기

코드가 생성되었다고 끝이 아닙니다. 강사는 곧바로 **개발 서버를 띄워** 생성된 페이지에 오류가 없는지 눈으로 확인합니다. AI가 만들어 준 결과물을 무작정 신뢰하기보다, 실제로 동작하는지 직접 검증하는 습관이
중요하기 때문입니다.

> "항상 작업이 끝나면 Git 커밋하는 것도 중요하지만, 이렇게 테스트하는 습관을 들이시면 좋을 것 같아요."

### 모바일 레이아웃 중앙 배치하기

페이지는 만들어졌지만, 기본 상태에서는 콘텐츠가 화면 전체로 퍼져 모바일 앱 같은 느낌이 나지 않습니다. 그래서 이벤트 페이지 레이아웃을 손봅니다. 뷰포트에 **`max-width`를 지정**해 콘텐츠 영역의 최대
너비를 모바일 사이즈로 제한하고, 이를 **화면 중앙에 배치**하는 작업입니다.

이때 막연하게 요청하기보다 **참고 이미지를 함께 제공**하는 것이 효과적입니다. 강의에서는 **와이즐리 쇼핑몰의 Mobile First 기반 UI/UX**를 스크린샷으로 캡처해 컨텍스트로 넘깁니다. AI가 어떤
모양을 만들어야 하는지 명확히 이해하도록 돕는 것입니다.

수정을 요청하면, 브라우저에서 의도한 대로 **모바일 사이즈로 중앙 정렬된 레이아웃**이 완성된 것을 확인할 수 있습니다.

### 타입 정의 및 인터페이스 설계

골격이 어느 정도 잡혔으니, 로드맵의 다음 단계인 **타입 정의 및 인터페이스 설계**로 넘어갑니다. 흐름은 앞서 익힌 Shrimp 워크플로우와 동일합니다.

1. `plan-task` 로 **태스크 2번(타입 정의)** 의 계획을 세웁니다.
2. `execute-task` 로 계획된 작업을 실행하되, 이번에는 **`nextjs-app-developer` 서브 에이전트**를 통해 처리합니다.

작업이 끝나면 `lib/types` 디렉터리에 타입이 체계적으로 정리되고 리팩토링됩니다.

| 타입 분류                 | 설명                   |
|-----------------------|----------------------|
| **핵심 도메인 타입**         | 이벤트·참여자 등 비즈니스 핵심 모델 |
| **공통 유틸 / API 처리 타입** | 응답·요청 등 API 통신 관련 타입 |
| **관리자 타입**            | 관리자 영역에서 사용하는 타입     |
| **공통 컴포넌트 타입**        | 재사용 컴포넌트의 props 등    |

### 코드 검사와 코드 리뷰

생성된 코드가 제대로 작성되었는지 **테스트로 에러 유무를 확인**한 뒤, 한 걸음 더 나아가 코드 리뷰까지 진행합니다. 강사는 특히 **개발 경험이 적을수록 중간중간 지속적인 코드 리뷰가 중요**하다고 강조합니다.

이때 서브 에이전트 두 개를 **동시에** 활용합니다. **`Next.js App Developer`** 와 **`Code Reviewer`** 를 함께 붙여 종합적인 리뷰를 받는 것입니다. 리뷰 결과로 미들웨어를
활용한 보안 강화 같은 권장 사항이 나오지만, 지금 단계에서 꼭 필요하지 않은 부분은 과감히 넘어갑니다.

> "그러면 이러한 사항은 추후에 차근차근 개선하면 되기 때문에,"

골격 단계에서 모든 것을 완벽하게 만들려 하기보다, **지금 필요한 만큼만 다지고 개선은 뒤로 미루는** 판단도 워크플로우의 일부입니다.

### 작업 완료 확인과 커밋

마지막으로 **Shrimp Task UI**를 통해 9개의 서브 태스크가 모두 완료되었는지 확인합니다. 그리고 변경 사항을 **의미 있는 그룹별로 묶어 Git 커밋**하며 한 사이클을 마무리합니다. 작업 단위가
깔끔하게 끊어지면, 나중에 히스토리를 되짚거나 문제를 추적하기도 훨씬 수월합니다.

### 정리하며

이번 챕터에서 진행한 골격 구축의 전체 흐름을 정리하면 다음과 같습니다.

1. `execute-task` 로 페이지를 빈 껍데기로 **일괄 생성**하고(18개 페이지),
2. 개발 서버를 띄워 **직접 동작을 확인**한 뒤,
3. 참고 이미지를 활용해 **모바일 레이아웃을 중앙 배치**하고,
4. `plan-task` → `execute-task` 로 **타입을 정의**한 다음,
5. 두 개의 서브 에이전트로 **코드 리뷰**를 받고,
6. 완료를 확인한 뒤 **그룹별로 커밋**했습니다.

핵심은 'AI에게 맡기고 끝'이 아니라, **생성 → 확인 → 보완 → 리뷰 → 커밋**으로 이어지는 작은 사이클을 꾸준히 반복했다는 점입니다. 이렇게 Phase 1, 즉 애플리케이션 골격 구축이 마무리되었습니다.
다음 챕터에서는 이 골격 위에 더미 데이터를 얹어 가며 본격적으로 UI/UX를 완성해 보겠습니다.

## UI/UX 구현: 개발 워크프로우 익숙해지기

골격이 갖춰졌으니, 이제는 그 빈 껍데기에 **실제 화면을 채워 넣을** 차례입니다. 이번 챕터에서는 공통 컴포넌트부터 시작해 주최자·참여자 모바일 화면, 그리고 관리자 데스크톱 페이지까지 UI/UX를 차례로
완성합니다. 그 과정에서 반복적으로 등장하는 **"계획하고 구현하고"** 라는 AI 개발 워크플로우의 리듬에 익숙해지는 것이 이번 시간의 진짜 목표입니다.

> "계획하고 구현하고, 계획하고 구현하고."

### 반복되는 리듬: clear → plan → execute

앞 챕터에서도 보았지만, 이번 챕터의 모든 작업은 **동일한 3박자**로 진행됩니다.

| 단계     | 명령             | 역할                     |
|--------|----------------|------------------------|
| **정리** | `/clear`       | 이전 대화 맥락을 비워 새 작업에 집중  |
| **계획** | `plan task`    | 로드맵 기준으로 작업을 계획·분할     |
| **실행** | `execute task` | 서브 에이전트를 통해 계획을 코드로 구현 |

이 리듬을 몸에 익히면, 어떤 작업이 와도 **"비우고 → 계획하고 → 실행하고"** 의 흐름으로 자연스럽게 풀어 갈 수 있습니다. 이번 챕터는 사실상 이 사이클을 화면 단위로 네 번 반복하는 과정입니다.

### 공통 컴포넌트 구현

가장 먼저 여러 페이지에서 재사용할 **공통 컴포넌트**부터 만듭니다. `shrimp-task-manager`와 `Next.js` 서브 에이전트를 활용해 작업을 계획하고 실행하면, 다음 세 가지가 한 번에
정리됩니다.

- **`shadcn/ui` 컴포넌트 설치** — 검증된 UI 컴포넌트 라이브러리 기반
- **공통 컴포넌트 구현** — 버튼·카드 등 반복 사용되는 요소
- **더미 데이터 세팅** — 실제 데이터 연동 전, 화면을 채울 가짜 데이터

공통 컴포넌트와 더미 데이터를 먼저 깔아 두면, 이후 페이지 작업에서 매번 바닥부터 만들 필요 없이 **속도가 붙습니다.**

### 주최자 모바일 UI/UX 완성

다음은 로드맵에 따라 **주최자(주체자)용 모바일 화면**을 완성합니다. 이벤트 페이지 유틸리티를 먼저 작성하고, 이벤트 목록·상세 페이지, 이벤트 생성·수정 폼, 프로필 페이지를 `plan task`로 계획한 뒤
`Next.js` 서브 에이전트로 구현합니다.

이 과정에서 **실전에서 흔히 마주치는 오류**도 만납니다. 외부 이미지를 불러올 때 `next.config.js`에 **이미지 도메인을 등록하지 않아** 로딩 오류가 발생하는 것입니다. 강사는 오류 메시지를
분석하고 도메인을 등록해 해결하는 과정을 그대로 보여줍니다.

> **Tip.** Next.js에서 외부 이미지를 사용하려면 `next.config.js`의 이미지 `domains`(또는 `remotePatterns`) 설정에 해당 호스트를 등록해야 합니다. 등록하지 않으면
> 이미지가 깨집니다.

오류를 바로잡고 나면 이벤트 목록, 상세 페이지, 새 이벤트 생성 폼, 프로필 페이지 UI가 모두 정상적으로 동작합니다. AI에게 맡기되, **에러는 직접 읽고 해결하는** 자세가 핵심입니다.

### 참여자 모바일 UI/UX 완성

이어서 초대받은 **참여자용 화면**을 만듭니다. 마찬가지로 `clear` 후 `plan task`로 계획하고, `Next.js` 앱 개발 서브 에이전트를 통해 3개의 태스크를 실행합니다. 여기서도
`next.config.js` 도메인 오류를 한 번 더 잡고, **초대 페이지**와 **참여 코드 유효성 검사** 기능까지 확인한 뒤 그룹별로 커밋합니다.

이 대목에서 강사는 AI 시대에도 **기본기 학습이 왜 중요한지**를 힘주어 강조합니다.

> "왜냐하면 아무리 정말 좋은 AI 기술이 있다고 하더라도, 내가 0이라면 결과는 0이겠죠?"

> "아는 만큼 보인다고, 더 나은 결과물을 얻고 싶으신 분들은 조금씩이라도 학습하는 걸 권장 드립니다."

권장하는 학습 순서는 **HTML → CSS → JavaScript → React → Next.js** 입니다. AI가 코드를 대신 짜 주더라도, 그 결과물을 읽고 판단할 수 있는 눈은 결국 본인의 기본기에서
나옵니다.

### 관리자 데스크톱 페이지 UI 구현

마지막은 모바일이 아닌 **데스크톱 기반 관리자 페이지**입니다. `plan task`로 패키지 설치, 더미 데이터 생성, 로그인 페이지 구성, 관리자 UI 제작을 계획합니다. 이번에는 정적 마크업 스타일링에 특화된
**`UI markup specialist`** 서브 에이전트를 활용합니다.

> "너무 단조롭게 구현을 하는 것보다 다양한 방법을 보여드리는 게 좋을 것 같아서요."

작업을 마치고 보니 **로그인 페이지에 관리자 전용 좌측 사이드바 메뉴가 노출되는 문제**가 발견됩니다. 로그인 전에는 메뉴가 보여선 안 되겠죠. 강사는 이를 다음 흐름으로 해결합니다.

1. **Playwright MCP**로 페이지 상태를 테스트하며 문제를 재현·확인합니다.
2. `plan task`로 **로그인 페이지를 사이드바 레이아웃에서 분리**하는 디렉터리 구조 변경을 계획합니다.
3. 계획을 실행한 뒤 개발 서버를 재실행해, 로그인 페이지에서 좌측 메뉴가 사라진 것을 확인합니다.

이는 단순한 UI 버그 수정을 넘어, **관리자 페이지는 로그인해야만 접근 가능하고 로그인 화면은 별도 레이아웃으로 분리해야 한다**는 보안·구조 원칙을 함께 보여 주는 사례입니다.

### 정리하며

이번 챕터에서는 동일한 워크플로우를 화면 단위로 반복하며 UI 전체를 완성했습니다.

1. **공통 컴포넌트**(shadcn/ui·더미 데이터)로 기반을 다지고,
2. **주최자 모바일 UI**를 만들며 `next.config.js` 이미지 오류를 해결하고,
3. **참여자 모바일 UI**로 초대·참여 코드 흐름을 완성한 뒤,
4. **관리자 데스크톱 UI**를 만들고 Playwright MCP로 레이아웃 문제까지 바로잡았습니다.

두 가지를 기억하면 좋겠습니다. 하나는 **`clear → plan → execute`의 리듬을 반복**하며 복잡한 작업을 잘게 위임했다는 점, 다른 하나는 그 와중에도 **에러를 직접 읽고 기본기를 다지는 자세**
를 놓지 않았다는 점입니다. AI는 0을 100으로 만들어 주지 않습니다. 내가 1이라도 알고 있을 때, AI는 그 결과를 몇 배로 키워 줍니다. 이렇게 웹 애플리케이션의 UI 구현이 모두 마무리되었습니다.

## 테마 스타일 커스텀하기: tweakcn

UI 구현을 마치고 나면 한 가지 아쉬움이 남습니다. `shadcn/ui`는 깔끔하고 완성도 높은 디자인을 제공하지만, 그만큼 **어디서 본 듯한 똑같은 느낌**을 주기 쉽습니다. 이번 챕터에서는 `shadcn`
기반 테마를 손쉽게 바꿔 주는 도구 **tweakcn**으로 우리 앱에 개성을 입혀 봅니다.

> "모든 사이트가 똑같아 보인다는 단점이 있어요."

### 왜 tweakcn이 필요한가

테마를 바꾸려면 결국 CSS 변수를 손봐야 합니다. 그런데 이 작업을 **직접 하면 시간이 오래 걸리고**, 그렇다고 **AI에게 맡기면 디자인 일관성이 깨지기** 쉽습니다. 색 하나를 바꿨더니 여기저기 톤이
어긋나는 경험, 한 번쯤 해보셨을 겁니다.

> "시간도 조금 소요되고 AI를 활용해서 커스터마이징 했다고 하더라도 일관성이 깨질 수 있죠."

tweakcn은 바로 이 지점을 해결합니다. **Tailwind의 규칙을 준수하면서** 테마를 일관성 있게 커스터마이징할 수 있게 해 주는 도구입니다.

> "정말 간편하게 `Tailwind` 룰을 준수하면서 우리 테마를 커스터마이징 할 수 있는 그러한 도구라고 보시면 됩니다."

### 기본 사용법: 에디터에서 테마 고르기

tweakcn 공식 문서에서 스타일 커스터마이징 버튼을 누르면 **에디터 UI**가 나타납니다. 사용 흐름은 직관적입니다.

- 좌측에서 **프리셋 테마를 선택**하면 전체 스타일이 즉시 바뀝니다.
- `Primary`, `Secondary` 등 **개별 컬러**를 따로 조정할 수 있습니다.
- 마음에 안 들면 **리셋**으로 초기 값으로 되돌립니다.
- 마음에 들면 우측의 **코드(Code)** 버튼에서 설치 명령어를 얻습니다.

문서만 보던 화면이 클릭 몇 번으로 전혀 다른 분위기로 바뀌는 모습은 꽤 인상적입니다.

> "정말 멋지지 않나요?"

### 프로젝트에 적용하기

에디터에서 얻은 설치 명령어를 프로젝트 터미널에 붙여 넣으면 됩니다. 다만 **반드시 먼저 해야 할 일**이 있습니다.

> **주의.** 명령어를 실행하기 전에 **현재 소스코드를 커밋**해 두세요. tweakcn 설치는 글로벌 CSS 변수를 덮어쓰기 때문에, 결과가 마음에 들지 않을 때 깔끔하게 되돌리려면 커밋이 안전망이 됩니다.

명령어를 실행하면 기존 CSS 설정을 덮어쓸지 묻고, `Y`를 입력하면 글로벌 CSS 변수가 교체됩니다. 이후 UI를 확인하면 선택한 테마가 적용되어 있고, 마음에 들지 않으면 변경 내용을 취소(revert)해 이전
화면으로 돌아갈 수 있습니다.

### 복병: Tailwind 버전 충돌

'버블검' 같은 다른 테마를 적용하다 보면, **스타일이 전혀 먹지 않는** 상황을 만날 수 있습니다. 원인은 **Tailwind 버전 차이**입니다.

| 구분             | Tailwind v3             | Tailwind v4   |
|----------------|-------------------------|---------------|
| **설정 파일**      | `tailwind.config.js` 사용 | 사용하지 않음       |
| **컬러 표현**      | 기존 방식                   | `oklch` 컬러 사용 |
| **tweakcn 호환** | 충돌 가능                   | 최신 tweakcn 기준 |

tweakcn은 **Tailwind 최신 버전(v4)** 을 기준으로 코드를 생성하는데, 프로젝트가 **v3**이면 충돌이 납니다. 특히 Supabase 템플릿으로 프로젝트를 세팅한 경우 이 문제가 잘
드러납니다. (참고로 이전에 만든 견적서 조회 시스템에서 테스트했을 때는 정상 적용되는 것을 확인했습니다.)

### 해결: Tailwind v4로 업그레이드

해결책은 프로젝트의 Tailwind를 **v4로 업그레이드**하는 것입니다. 흐름은 다음과 같습니다.

1. Tailwind가 제공하는 **공식 업그레이드 도구**로 최신 안정화 버전(v4.1)으로 올립니다.
2. v4에서는 쓰지 않는 **`tailwind.config.js` 파일을 제거**합니다.
3. `package.json`에서 버전이 바뀐 것을 확인합니다.
4. `shadcn` 테마를 **다시 설치**해 '버블검', '레트로', 커스텀 '인디고' 등 원하는 테마를 적용합니다.

업그레이드 이후에는 앞서 막혔던 테마들이 정상적으로 입혀집니다.

### 더 디테일하게: Toggle Inspector

프리셋만으로 부족하다면 색상·타이포그래피를 직접 조정하고, 다양한 컴포넌트로 미리보기를 확인할 수 있습니다. 더 세밀하게 다루고 싶다면 **`Toggle Inspector`** 를 켜 보세요. UI 요소를 직접
클릭해 설정을 바꿀 수 있어, 예를 들어 **카드의 background를 클릭 한 번으로 변경**하면 즉시 화면에 반영됩니다.

커스텀한 테마는 Sign in 후 **Save** 버튼으로 이름을 붙여 저장할 수 있고(강사도 이름 짓기엔 약했지만요), 저장된 테마는 설치 명령어로 언제든 프로젝트에 가져올 수 있습니다.

> "마땅히 생각나는 이름이 없네요."

### 정리하며

이번 챕터의 핵심은 **"일관성을 지키면서 빠르게 커스터마이징한다"** 입니다.

- `shadcn/ui`의 획일적인 디자인을 **tweakcn**으로 손쉽게 개성 있게 바꿨고,
- 마크업에서 **인라인 스타일을 직접 건드리는 대신**, 테마 단위로 일관성 있게 조정했으며,
- 도중에 만난 **Tailwind v3 ↔ v4 충돌**을 버전 업그레이드로 해결했습니다.

디자인에 자신이 없을수록 이런 테마 에디터의 힘은 더 크게 다가옵니다. 코드를 직접 만지지 않고도 Tailwind 규칙 안에서 완성도 있는 디자인을 입힐 수 있으니, 1인 개발자에게는 특히 든든한 도구입니다.

## UI/UX 개선: plan mode

UI를 한 번 완성했다고 끝이 아닙니다. 막상 써보면 **"여기는 좀 고치고 싶은데"** 싶은 부분이 계속 눈에 들어옵니다. 이번 챕터에서는 그동안 일부러 미뤄 두었던 자잘한 UI/UX 개선들을, Claude
Code의 **Plan 모드**를 활용해 체계적으로 다듬어 봅니다.

> "제가 중간중간에 UI나 UX는 개선하고 싶은 게 많았거든요. 그런데 일부러 개선을 안 했어요."

### 무엇을 개선할 것인가

이번에 손볼 항목은 사용하다 보면 거슬리는 **세 가지 디테일**입니다.

| 개선 항목          | 문제              | 개선 방향       |
|----------------|-----------------|-------------|
| **로그인 페이지 언어** | 영어로 표시됨         | 한국어로 변경     |
| **로그인 후 이동**   | 엉뚱한 곳으로 가거나 머무름 | 홈으로 리디렉션    |
| **홈 화면 레이아웃**  | 모바일 최적화 미적용     | 모바일 레이아웃 적용 |

하나하나는 사소해 보이지만, 이런 디테일이 모여 앱의 완성도를 결정합니다.

### 바로 고치지 않고 Plan 모드로

여기서 이 챕터의 핵심이 등장합니다. 개선 요청을 받았다고 **곧바로 코드를 수정하게 두지 않는** 것입니다. 먼저 **Plan 모드**로 전환해 작업 계획부터 세우게 합니다.

> "그리고 바로 개선을 하지 않고 작업 계획을 세워야 하는데요. Plan 모드로 진행하도록 할게요."

무작정 코드를 건드리기 시작하면 의도와 다른 방향으로 흘러가거나, 엉뚱한 파일을 수정해 버리기 쉽습니다. Plan 모드는 **"무엇을, 어떻게 바꿀지"를 먼저 합의**하고 들어가는 안전장치인 셈입니다. 작은
개선일수록 오히려 이 습관이 빛을 발합니다.

### 계획 검토 후 Bypass permissions로 실행

Plan 모드가 세워 준 계획을 **눈으로 검토**하고, 의도대로 잘 잡혔는지 확인합니다. 계획이 마음에 들면 그때 **Bypass permissions 모드**로 전환해 실제 개선 작업을 한 번에 진행합니다.

즉, 흐름은 이렇게 정리됩니다.

1. **개선 요청** — 고치고 싶은 점을 자연어로 전달
2. **Plan 모드** — 곧바로 수정하지 않고 작업 계획 수립
3. **계획 검토** — 사용하지 않는 폴더 정리 등 계획 확인
4. **Bypass permissions** — 검토된 계획을 막힘 없이 실행

계획을 한 번 거른 뒤 실행하기 때문에, 속도와 안전성을 동시에 챙길 수 있습니다.

### 결과 확인과 추가 개선

작업이 끝나면 브라우저에서 결과를 확인합니다. 홈 화면에 **모바일 레이아웃이 적용**되고, 로그인 페이지가 **한국어로 표시**되며, 로그인 후 **홈으로 리디렉션**되는 것을 볼 수 있습니다.

물론 한 번에 완벽하진 않습니다. 일부 깨진 UI가 남아 있다면, **스크린샷을 찍어 컨텍스트로 제공**하며 추가 개선을 요청합니다. 이렇게 한 번 더 다듬으면 이전과 확연히 달라진 홈 UI가 완성됩니다.

이 과정에서 강사는 **가능한 한 코드를 직접 건드리지 않는다**는 원칙을 다시 한번 강조합니다.

> "웬만하면 제가 직접 코드는 안 건드리고 있어요. 왜냐하면 수강생분들 중에 개발 경험이 없으신 분들도 계시기 때문에, 그래서 제가 거의 다 Claude Code를 통해서 진행하고 있죠."

직접 코드를 수정할 줄 알더라도, **개선 의도를 명확히 전달하고 결과를 검증하는** 워크플로우만으로 충분히 완성도를 끌어올릴 수 있다는 것을 보여 줍니다.

### 정리하며

이번 챕터는 짧지만 메시지는 분명합니다.

> **아무리 작은 개선이라도, 곧바로 코드를 고치기보다 Plan 모드로 계획을 먼저 세우자.**

- 미뤄 두었던 **언어·리디렉션·모바일 레이아웃** 디테일을 다듬었고,
- **Plan 모드 → 계획 검토 → Bypass permissions**의 흐름으로 안전하게 실행했으며,
- 깨진 부분은 **스크린샷으로 추가 개선**을 요청해 마무리했습니다.

작은 수정 하나에도 '계획 먼저'라는 원칙을 지키는 것 — 그것이 AI와 협업하면서도 결과를 통제 가능하게 만드는 핵심 습관입니다.

## 작업 구현: 기능 구현 1 (+Supabase MCP)

지금까지는 더미 데이터로 화면만 채워 왔습니다. 이제는 진짜 **데이터베이스를 붙이고 핵심 기능을 구현**할 차례입니다. 이번 챕터에서는 **Supabase MCP**를 활용해 DB 스키마를 설계하고, 사용자
인증·권한 관리부터 로그아웃 기능까지 만들어 봅니다. 정적인 껍데기가 실제로 동작하는 앱으로 바뀌는 전환점입니다.

### 핵심 워크플로우: 탐색 → 계획 → 구현 → 커밋

이번 챕터를 관통하는 가장 중요한 메시지부터 짚고 갑니다. 강사가 **반복해서 강조하는 4단계 개발 워크플로우**입니다.

> "우리가 어떠한 프로덕트나 어떠한 기능을 구현할 때 탐색을 통한 자료 수집, 계획, 구현, 마지막 커밋을 항상 고려하시는 걸 권장 드릴게요."

| 단계     | 내용                        |
|--------|---------------------------|
| **탐색** | 자료 수집 — 무엇이 필요한지 정보부터 모은다 |
| **계획** | 수집한 정보로 작업 계획을 세운다        |
| **구현** | 계획대로 코드를 작성한다             |
| **커밋** | 완료된 작업을 의미 단위로 기록한다       |

앞 챕터들의 `plan → execute`보다 한 걸음 앞에 **'탐색'** 이 붙은 것이 핵심입니다. 데이터베이스처럼 한번 잘못 설계하면 되돌리기 어려운 작업일수록, 곧바로 계획에 들어가지 않고 **충분히 자료를
수집한 뒤** 시작하는 것이 중요합니다.

### 탐색: Supabase MCP로 자료 수집

먼저 DB 스키마와 Supabase 초기 설정(로드맵의 Task 7)을 잘 수행하기 위한 **정보 수집**부터 합니다. 이때 **Next.js·Supabase 전문 서브 에이전트**가 **Supabase MCP**를
활용해, 현재 데이터베이스 정보를 파악하고 스키마를 어떻게 구축해야 할지 요약해 줍니다.

> "그러면 이러한 Subagent가 어떻게 데이터베이스 스키마를 구축해야 하는지 보시는 것처럼 Supabase MCP를 활용해서 보다 효과적인 정보로 요약을 해 줄 거예요."

이렇게 수집된 정보와 로드맵 파일을 바탕으로, **Shrimp Task Manager**로 정교한 계획을 세웁니다. 막연한 추측이 아니라 **실제 DB 상태에 근거한 계획**이라는 점이 차이를 만듭니다.

### 계획: DB 스키마 설계

수집한 정보를 토대로 데이터베이스 구조를 **순차적으로** 설계합니다.

1. **유저(User) 테이블** — 사용자 정보 저장
2. **이벤트(Event) 테이블** — 이벤트 데이터 저장
3. **스토리지 서비스** — 이벤트 커버 이미지 등 파일 저장

한꺼번에 모든 것을 만들기보다, 핵심 테이블부터 차근차근 쌓아 올리는 방식입니다. 참고로 유출된 비밀번호 사용을 막는 보안 기능은 **Supabase Pro Plan 제약**이 있어, 이번에는 해당 부분을 감안하고
진행합니다.

### 구현: 사용자 인증과 권한 관리

스키마가 준비되면 본격적으로 **인증·권한 기능**을 구현합니다. 사용자 유형별로 접근 권한을 명확히 나누는 것이 핵심입니다.

| 사용자 유형      | 어드민 대시보드 접근 시 동작 |
|-------------|------------------|
| **일반 사용자**  | 홈으로 리다이렉션        |
| **미인증 사용자** | 로그인 페이지로 이동      |
| **관리자**     | 대시보드 정상 접근       |

관리자 역할은 **이메일 변경 쿼리**로 부여하고, 테스트로 정상 작동을 확인합니다. 그런데 여기서 **역할을 부여했는데도 대시보드 접속이 안 되는 문제**가 발생합니다. 강사는 이를 **로그를 추가하고
Supabase MCP로 상태를 확인**하며 디버깅해 해결합니다. AI가 코드를 짜 주더라도, 문제가 생기면 **로그를 찍어 원인을 추적하는** 기본기는 여전히 유효합니다.

### 서브 에이전트가 말을 안 들을 때

기능 구현 중에는 **서브 에이전트가 요청대로 실행되지 않는** 경우도 생깁니다. 강사는 이럴 때의 대처법을 공유합니다.

> "간혹가다가 이렇게 요청을 했음에도 불구하고 Subagent가 실행이 안 될 때도 있어요. 그럴 때는 과감히 ESC로 작업 중지를 해주시고요. 명확하게 Subagent를 사용해달라고 프롬프트를 추가하시면 돼요."

즉, **ESC로 멈추고 → 서브 에이전트를 쓰라고 명시적으로 다시 요청**하면 됩니다. AI 도구가 늘 한 번에 의도대로 움직이는 것은 아니므로, 이런 대처법을 알아 두면 흐름이 끊기지 않습니다.

### UI 마무리: 로그아웃 기능

마지막으로 관리자·사용자 UI에 **로그아웃 버튼**이 있는지 확인하고, 없으면 추가합니다. 구현 후 프로필 화면에서 로그아웃 버튼을 누르면 **로그인 페이지로 이동**하는 것까지 테스트하며 마무리합니다.

### 정리하며

이번 챕터에서는 더미 데이터를 넘어 **실제 데이터베이스와 인증 기능**을 붙였습니다.

1. **탐색** — Supabase MCP로 DB 정보를 수집하고,
2. **계획** — 유저·이벤트 테이블, 스토리지를 순차 설계한 뒤,
3. **구현** — 사용자 유형별 권한과 로그아웃 기능을 만들고,
4. **디버깅** — 로그와 Supabase MCP로 권한 문제를 해결했습니다.

가장 기억할 점은 **'탐색 → 계획 → 구현 → 커밋'** 이라는 4단계 리듬입니다. 특히 데이터베이스처럼 신중함이 필요한 작업에서는, 곧바로 만들기보다 **MCP로 실제 상태를 탐색한 뒤** 계획을 세우는 것이
결과의 질을 좌우합니다. 다음 챕터에서는 이 기반 위에 핵심 기능 구현을 이어 갑니다.

## 작업 구현: 기능 구현 2

앞 챕터에서 DB 스키마와 인증의 기초를 닦았습니다. 이번에는 그 위에서 **사용자 관리 기능을 한층 강화**하고, 이미지 저장을 위한 **스토리지 버킷**까지 만들어 데이터베이스·Supabase 설정을
마무리합니다. 짧지만, **계획을 상황에 맞게 유연하게 수정하는** 중요한 장면이 담긴 챕터입니다.

### 계획을 그대로 따르지 않는다: profiles 테이블 강화

원래 다음 작업은 **`users` 테이블을 새로 생성**하는 것이었습니다. 그런데 프로젝트에는 이미 **`profiles` 테이블이 존재**했습니다. 강사는 여기서 계획을 그대로 따르지 않고, 방향을 바꿉니다.

> "제가 이러한 작업 계획을 새로운 `users` 테이블을 생성하지 말고, 기존 `profiles` 테이블을 강화해 달라고 그렇게 작업 계획을 수정했습니다."

이것이 이번 챕터의 핵심 교훈입니다. 계획은 **고정된 정답이 아니라 현재 상황에 맞춰 조정하는 대상**입니다. 이미 있는 테이블을 두고 비슷한 테이블을 또 만들면 데이터가 분산되고 관리만 복잡해지죠. **새로
만들기보다 기존 자산을 확장**하는 편이 효율적이라는 판단입니다.

### execute task로 테이블 강화하기

방향을 정했으니 **`execute task`** 로 Next.js·Supabase 전문 서브 에이전트에게 작업을 맡깁니다. 이때 추가되는 것들은 다음과 같습니다.

| 작업            | 내용                           |
|---------------|------------------------------|
| **기존 테이블 분석** | 현재 `profiles` 구조 파악          |
| **컬럼 추가**     | `profiles` 테이블에 필요한 필드 확장    |
| **자동화 트리거**   | 데이터 무결성을 위한 트리거 설정           |
| **RLS 정책 개선** | Row Level Security로 접근 보안 강화 |

참고로 일부 보안 기능은 **Pro Plan 전용**이라 비활성화 상태로 둡니다.

> "이러한 건 Pro Plan 기능이기 때문에 해당 기능은 비활성화로 그대로 두도록 하고요."

### 남은 작업 일괄 처리와 서브 에이전트 재요청

이어서 `/clear`로 맥락을 정리한 뒤, 남은 작업을 **한 번에 일괄 진행**합니다. 그런데 여기서도 앞 챕터에서 본 것과 같은 상황이 재현됩니다. **서브 에이전트가 제대로 인식되지 않는** 것입니다.

> "왜냐하면 이러한 서브 에이전트 인식이 안 된 것 같아요."

이럴 때의 대처는 동일합니다. **서브 에이전트를 사용하도록 명확히 재요청**하면 됩니다. 다시 요청하자 서브 에이전트가 정상 작동하며 남은 작업을 성공적으로 완료합니다. AI 도구가 한 번에 안 먹힐 수 있다는 걸
알고 있으면, 당황하지 않고 흐름을 이어 갈 수 있습니다.

### 스토리지 버킷 생성 확인

작업이 끝나면 여러 테이블 작업과 함께 **`event_cover`** 라는 새로운 **스토리지 버킷**이 생성된 것을 확인할 수 있습니다. 이 버킷은 다음을 지원합니다.

- **폴더 기반 파일 관리** — 체계적인 파일 구조
- **이미지 파일 업로드** — 이벤트 커버 이미지 저장

이제 이 버킷을 활용하면 이벤트 커버 이미지를 업로드하는 기능을 붙일 수 있고, 이로써 **데이터베이스 스키마와 Supabase 설정이 완료**됩니다.

### 정리하며

짧은 챕터였지만 메시지는 묵직합니다.

> **계획은 절대적이지 않다. 현재 상황에 맞게 유연하게 수정하는 것도 실력이다.**

- `users` 테이블을 새로 만드는 대신 **기존 `profiles` 테이블을 강화**했고,
- **자동화 트리거·RLS 정책**으로 데이터 무결성과 보안을 다졌으며,
- **`event_cover` 스토리지 버킷**까지 만들어 Supabase 설정을 마무리했습니다.

AI에게 계획을 맡기되, 그 계획이 현재 코드베이스와 맞는지 **사람이 판단하고 조정**하는 것 — 이것이 AI와 협업하는 개발자의 진짜 역할입니다. 이렇게 데이터베이스 기반이 완성되었으니, 이제 그 위에 실제
기능을 얹어 갈 준비가 끝났습니다.

## 작업 계획 및 구현: 인증 및 프로필

데이터베이스 기반을 다졌으니, 이번에는 그 위에서 **인증 시스템과 프로필 관리**를 완성합니다. 회원가입·로그인 흐름을 만들고, 실제로 테스트하며 오류를 잡아 나가는 과정입니다. 그리고 마지막에는 **"오류를 언제
고칠 것인가"** 라는, 실전에서 의외로 중요한 노하우 하나를 배웁니다.

### 이미 90%는 되어 있다: 개선 중심 계획

`plan task`로 인증 작업을 계획하고 분석해 보니, 흥미로운 사실이 드러납니다. **이전 실습에서 이미 90%가 구현**되어 있던 것입니다.

> "그런데 이러한 목록을 살펴보면 우리가 이전 시간에 실습을 하면서 모두 구현한 것들이에요."

그렇다고 그냥 넘어가지 않습니다. **개선 사항**을 중심으로 다시 계획을 세웁니다. 남은 작업은 **프로필 자동 생성, 관리자 인증, 테스트** 정도입니다.

> "하지만 개선 사항이 있을 수 있기 때문에 이러한 것도 Shrimp Task Manager를 통해서 계획을 생성하고 구현을 해보도록 할게요."

이미 되어 있는 것도 한 번 더 점검하고 다듬는 자세 — 이것이 완성도의 차이를 만듭니다.

### 단계별 계획: 8개 작업

분석 결과 작업은 **8개로 분할**되고, 세 개의 Phase로 정리됩니다.

| Phase       | 내용                    |
|-------------|-----------------------|
| **Phase 1** | 데이터베이스 작업             |
| **Phase 2** | 인증 로직 완성 및 UX 개선      |
| **Phase 3** | Playwright MCP 기반 테스트 |

특히 마지막 Phase에 **테스트가 계획에 포함**되어 있다는 점이 눈에 띕니다. 구현과 검증을 한 흐름으로 묶는 것이죠. 계획이 서면 `execute task`로 모든 작업을 한 번에 실행합니다.

### 꼼꼼한 테스트: 회원가입과 DB 연동 확인

구현 후에는 **직접 테스트**합니다. 구글 소셜 로그인으로 회원가입을 진행하고, 데이터가 제대로 들어갔는지 Supabase에서 두 곳을 확인합니다.

1. **Authentication의 User 테이블** — 새로고침 시 가입 데이터가 등록됨
2. **Table Editor의 프로필 테이블** — **데이터베이스 트리거**에 의해 프로필 데이터가 자동 생성됨

앞 챕터에서 설정한 자동화 트리거가 실제로 동작해, 회원가입만 했는데 프로필이 알아서 만들어지는 것을 눈으로 확인하는 순간입니다.

### 테스트에서 드러난 두 가지 오류

꼼꼼히 테스트하니 **숨어 있던 문제 두 가지**가 드러납니다.

- **중복 회원가입 허용** — 이미 가입된 이메일로 다시 가입하면 막아야 하는데, 그냥 가입이 완료됨
- **영어 오류 메시지** — 이메일 인증 전 로그인 시도 시, "로그인할 수 없다"는 메시지가 **영어로** 표시됨

테스트를 대충 했다면 놓쳤을 문제들입니다. 강사는 이를 `plan task`로 수정 계획을 세운 뒤, **중복 가입을 막고 / 실패 원인을 한국어로 표시**하도록 개선합니다. 바이패스 퍼미션 모드로 수정을 적용하고
다시 테스트하면, 이미 가입된 이메일에 대해 **한국어 오류 메시지**가 정상적으로 표시됩니다.

### 핵심 팁: 오류에도 우선순위가 있다

이번 챕터에서 가장 실용적인 부분입니다. 강사는 기능 구현 중 만나는 오류를 **두 종류로 구분**하라고 조언합니다.

> "우리가 기능을 구현하면서 오류가 발생한다면 반드시 지금 해결해야 하는 오류가 있고요. 그게 아니라면 나중에 해결해야 하는 오류가 있어요."

| 구분         | 예시                      | 처리 방식                   |
|------------|-------------------------|-------------------------|
| **즉시 해결**  | 중복 가입 허용 같은 기능 결함       | 바로 수정                   |
| **나중에 해결** | 영어 오류 메시지, 사소한 UX/UI 이슈 | `ISSUES` 파일에 기록 후 추후 처리 |

핵심은 **`ISSUES` 파일**을 만들어 급하지 않은 오류를 모아 두는 것입니다. 이렇게 모아 두면 나중에 **한꺼번에 처리**하거나, **공통 컴포넌트·모듈로 묶어** 효율적으로 개선할 수 있습니다. 모든
오류를 그때그때 잡으려다 보면 정작 중요한 흐름이 끊기기 때문에, **우선순위를 나눠 관리**하는 것이 실전에서 훨씬 생산적입니다.

### 정리하며

이번 챕터에서는 인증·프로필 기능을 완성하며 다음을 경험했습니다.

1. 이미 구현된 부분도 **개선 중심으로 다시 계획**하고(8개 작업, 3 Phase),
2. **트리거 기반 프로필 자동 생성**과 DB 연동을 직접 확인했으며,
3. 테스트로 **중복 가입·영어 메시지 오류**를 발견해 수정하고,
4. 오류를 **즉시/나중**으로 나눠 `ISSUES` 파일로 관리하는 법을 배웠습니다.

가장 기억할 것은 **오류의 우선순위 관리**입니다. AI와 빠르게 개발할수록 자잘한 이슈가 쌓이기 마련인데, 이를 `ISSUES`로 모아 두고 흐름을 지키는 것 — 그것이 속도와 완성도를 동시에 챙기는 현실적인
전략입니다.

## 작업 계획 및 구현: 이벤트 CRUD 1

이제 이 앱의 **심장부**에 도착했습니다. 바로 **이벤트 CRUD**(생성·수정·삭제) 기능입니다. 이번 챕터는 유난히 오류가 많이 등장하는데, 그게 오히려 핵심입니다. AI와 함께 개발하다 보면 **예상치 못한
버그를 끊임없이 마주치고 해결**하게 되는데, 그 현실적인 과정을 가감 없이 보여 주기 때문입니다.

> "하지만 현업에서도 개발을 할 때 버그는 항상 존재합니다."

### 계획 단계에서 컨텍스트를 미리 주입하기

`plan_task`로 작업을 계획하기 전에, 강사는 **필요한 기술 컨텍스트를 먼저 추가**합니다. 이벤트 등록·수정에는 폼이 필요하고, 데이터 저장에는 서버 로직이 필요하기 때문입니다.

- **React Hook Form** — 폼 관리와 유효성 검사를 효과적으로 처리
- **Next.js 서버 액션(Server Action)** — 클라이언트에서 서버 로직을 직접 호출

> "이러한 걸 효과적으로 구현할 수 있도록 `React Hook Form`에 대한 컨텍스트를 이렇게 정확히 추가하도록 할게요."

계획 단계에서 **어떤 도구를 쓸지 미리 명시**해 두면, AI가 엉뚱한 방식으로 구현하는 것을 막고 의도한 방향으로 코드를 작성하게 만들 수 있습니다.

### 또 등장한 서브 에이전트 인식 문제

이번에도 익숙한 장면이 반복됩니다. `plan_task`를 요청했는데 **Shrimp Task Manager가 인식되지 않는** 것입니다. 대처법은 이제 우리도 압니다 — 작업을 취소하고 **명확하게 Shrimp
Task Manager를 쓰라고 다시 요청**합니다. 그러자 계획이 정상적으로 수립되어, 공통 기능 구현 → Supabase 정책 설정 → 이벤트 생성·삭제 UI 통합·테스트까지 잘 정의됩니다.

### 버그의 연속: 4개의 오류를 잡아 나가다

본격 구현에 들어가자 오류가 줄줄이 등장합니다. 이 챕터의 백미는 이 **오류들을 하나씩 해결해 나가는 과정**입니다.

| #     | 오류 증상                                 | 원인/해결                                   |
|-------|---------------------------------------|-----------------------------------------|
| **1** | 등록 성공 메시지는 뜨는데 **DB에 데이터가 안 들어감**     | 에러 전문가 역할 부여 → 디버깅 로그로 원인 추적·해결         |
| **2** | "사용자 프로필을 찾을 수 없습니다" 오류               | 오류 메시지 복사 → 단계별 해결 + Playwright MCP 테스트 |
| **3** | 이벤트 생성 중 **초대 코드 중복** 오류              | 콘솔 오류를 컨텍스트로 추가 → 재요청해 해결               |
| **4** | Server Action만 구현되고 **UI 버튼과 연결 안 됨** | 오류가 아니라, 작업을 잘게 쪼개며 생긴 정상적 중간 상태        |

특히 주목할 패턴이 있습니다. 오류가 나면 강사는 **Claude Code에게 "Next.js Supabase 에러 해결 전문가" 역할을 부여**하고, **오류 메시지나 콘솔 로그를 그대로 복사해 컨텍스트로 제공**
합니다. 막연히 "안 돼요"가 아니라, **구체적인 증거를 주고 단계별 해결을 요청**하는 것이 핵심입니다.

> "이러한 걸 해결하는 것도 모두 보여드리도록 하겠습니다."

### 4번은 버그가 아니다: 잘게 쪼갠 작업의 중간 상태

네 번째 항목은 따로 짚을 가치가 있습니다. Server Action(서버 로직)은 구현됐는데 **UI 버튼과 연결되지 않은** 상태를 만나는데, 이는 **오류가 아닙니다.** 작업을 잘게 분할하다 보니, 비즈니스
로직과 UI 연결이 **서로 다른 태스크로 나뉘어** 아직 뒷 작업이 안 됐을 뿐입니다.

이런 구분은 중요합니다. 모든 "동작 안 함"이 버그는 아니며, **계획을 세분화했을 때 자연스럽게 생기는 중간 단계**일 수 있습니다. 이를 알아야 불필요한 디버깅에 시간을 쏟지 않습니다.

### 커버 이미지 업로드: 계획으로 빈틈 발견하기

마지막으로 **이벤트 커버 이미지 업로드** 기능을 붙입니다. 여기서도 `plan` 모드가 빛을 발합니다. 계획을 세우다 보니 **폼에 URL 입력 필드만 있고 정작 파일 업로드 UI가 없다**는 빈틈이 드러납니다.

곧바로 파일 업로드 UI를 추가하고, 이미지를 올린 뒤 이벤트를 생성하면 — Supabase 스토리지의 **`event_cover` 버킷에 이미지가 정상 저장**되고, 이벤트 테이블에도 커버 이미지 URL이 잘
기재됩니다. 앞 챕터에서 만들어 둔 스토리지 버킷이 마침내 제 역할을 하는 순간입니다.

### 작업 검증: list_task와 verify_task

오류 해결 사이사이, 강사는 작업 상태도 꼼꼼히 관리합니다. `/clear` 후 **`list_task`로 남은 작업을 확인**하고, 완료 처리되지 않은 작업을 업데이트하며, **`verify_task`로 작업을
검증**합니다. 구현에만 매몰되지 않고 **태스크의 진행 상태를 계속 동기화**하는 습관입니다.

### 정리하며

이번 챕터는 **오류와 함께 가는 개발**의 진짜 모습을 보여 주었습니다.

1. 계획 전에 **React Hook Form·서버 액션 컨텍스트를 주입**하고,
2. DB 미등록·프로필·초대 코드 중복 등 **연속된 오류를 단계별로 해결**했으며,
3. "UI 미연결"처럼 **버그가 아닌 중간 상태를 구분**하고,
4. `plan`으로 **파일 업로드 UI의 빈틈을 발견**해 커버 이미지 업로드를 완성했습니다.

핵심 노하우는 **오류를 다루는 방식**입니다. 에러 전문가 역할을 부여하고, **오류 메시지·콘솔 로그를 그대로 증거로 제공**하며 단계별 해결을 요청하는 것. 버그는 피할 수 없지만, 이렇게 접근하면 AI가 훨씬
정확하게 원인을 짚어 줍니다. 다음 챕터에서는 이벤트 CRUD의 나머지를 마저 구현합니다.

## 작업 계획 및 구현: 이벤트 CRUD 2

앞 챕터에서 이벤트 **생성(Create)** 을 완성했으니, 이번에는 나머지인 **수정(Update)과 삭제(Delete)** 를 구현해 CRUD를 마무리합니다. 이번 챕터에서 거듭 확인하게 될 두 가지 무기는 *
*Plan 모드**와 **오류 재현 프롬프트**입니다.

> "지난 시간에 이어서 계속해서 이벤트 수정 삭제 기능을 구현해 보도록 하겠습니다."

### 테스트를 위해 목록 UI부터

수정·삭제를 테스트하려면 먼저 **이벤트 목록**이 있어야 합니다. 그래서 `/clear` 후 "이벤트 목록을 구현해 주세요"라고 요청하되, 이번에도 **스크린샷을 첨부**해 원하는 모습을 명확히 전달합니다. 구현
후에는 익숙한 `next.config.js` **이미지 도메인 오류**를 다시 한번 등록으로 해결합니다. CRUD를 검증할 무대를 먼저 깔아 두는 셈입니다.

### 핵심 무기 ①: 오류를 '재현 시나리오'로 전달하기

이번 챕터에서 가장 실용적인 노하우입니다. 수정·삭제를 테스트하며 여러 오류를 만나는데, 강사는 오류를 해결할 때 **그 오류를 재현하는 방법을 프롬프트에 구체적으로 적어 줍니다.**

> "오류를 정확히 수정하고 싶다면 이러한 오류를 재현할 수 있는 방법을 프롬프트에 명확히 요청하는 것도 정말 좋은 방법 중 하나죠."

예를 들어 삭제 오류라면 이렇게 단계를 적습니다.

> **재현 시나리오 예시**
> 1. 이벤트 상세 페이지에 진입한다
> 2. 삭제 버튼을 클릭한다
> 3. 삭제 완료 후 목록으로 진입하면 오류가 발생한다

"안 돼요"가 아니라 **"이렇게 하면 이 오류가 난다"** 고 단계를 짚어 주면, AI가 문제를 정확히 재현하고 원인을 훨씬 잘 찾아냅니다. 막연한 신고보다 **재현 가능한 버그 리포트**가 강력한 이유와
같습니다.

### 핵심 무기 ②: Plan 모드의 문제 해결력

또 하나 반복되는 원칙은 **곧바로 고치지 말고 Plan 모드로 먼저 계획하라**는 것입니다. 강사는 이를 명확한 경험칙으로 정리합니다.

> "일반적으로 플랜 모드로 진행을 하게 되면, 바로 진행을 하는 것보다 훨씬 더 문제 해결 능력이 뛰어납니다."

그래서 이번 챕터의 오류 해결은 거의 모두 **Plan 모드로 계획 → Bypass permissions 모드로 실행**이라는 일관된 흐름을 따릅니다.

### 해결한 이슈들

수정·삭제를 완성하기까지 마주친 이슈와 해결 방식을 정리하면 다음과 같습니다.

| 이슈                         | 해결 방식                                          |
|----------------------------|------------------------------------------------|
| **상세 페이지 진입 시 목록으로 리다이렉션** | Playwright MCP로 현상 확인 + 테스트 계정 명시 → Plan 모드 해결 |
| **이벤트 삭제 시 오류**            | 삭제 시나리오를 단계별로 작성 → Plan 모드 계획 후 실행             |
| **수정 페이지 진입 시 리다이렉션**      | 테스트 계정 명시 + Playwright MCP로 재현 → 원인 파악·해결      |

특히 **리다이렉션 이슈**를 다룰 때는 **Playwright MCP에 테스트 계정을 명시**해 자동화 테스트를 돌립니다. 로그인이 필요한 페이지를 검증할 때, 어떤 계정으로 접속할지 알려 주면 AI가 실제 로그인
흐름까지 재현해 문제를 정확히 확인할 수 있습니다.

### 공통 UI 재활용: 등록 페이지와 수정 페이지

마지막으로 수정 페이지의 **커버 이미지 등록 UI를 파일 업로드 방식으로 개선**합니다. 이때 중요한 판단이 있습니다. 등록 페이지와 수정 페이지는 **거의 같은 폼**이므로, 새로 만들기보다 **공통 UI와
기능을 최대한 재활용**하도록 계획을 세우는 것입니다.

이렇게 하면 중복 코드를 줄이고, 한쪽을 고치면 양쪽이 함께 개선되는 구조가 됩니다. 작업을 마치면 **Git 커밋**으로 이벤트 CRUD 구현을 마무리합니다.

### 정리하며

이번 챕터로 이벤트 CRUD의 **수정·삭제까지 완성**하며 핵심 기능 한 덩어리를 끝냈습니다.

1. 테스트 무대로 **이벤트 목록 UI**를 먼저 만들고,
2. 오류는 **재현 시나리오를 프롬프트에 명시**해 정확히 해결했으며,
3. 모든 수정 작업을 **Plan 모드 → Bypass permissions** 흐름으로 진행하고,
4. 등록·수정 페이지의 **공통 UI를 재활용**해 효율적으로 마무리했습니다.

두 가지를 꼭 기억하면 좋겠습니다. **버그는 재현 방법과 함께 신고하라**, 그리고 **고치기 전에 Plan 모드로 계획하라.** 이 두 습관만으로도 AI와의 디버깅 효율은 확연히 달라집니다.

## 작업 계획 및 구현: 이벤트 참여

이벤트 CRUD를 끝냈으니, 이제 이 앱의 존재 이유에 해당하는 기능을 만들 차례입니다. 바로 **초대 링크를 통한 이벤트 참여**입니다. 이번 챕터의 두 축은 **'무엇을 만들지 않을지 정하는 판단'** 과, *
*로그인 리다이렉트 같은 사용자 플로우를 다듬는 과정**입니다.

### 만들지 않을 것부터 정한다: 오버스펙 걷어내기

본격 구현에 앞서 강사는 남은 작업을 점검하며, **굳이 만들 필요 없는 기능들을 과감히 덜어냅니다.**

> "이러한 기능은 현재 필요 없기 때문에 진행하지 않고,"

| 걷어낸 기능                  | 판단 이유            |
|-------------------------|------------------|
| **이벤트 상태 자동 관리**        | 현재 단계에서 불필요      |
| **Realtime 실시간 참여자 목록** | 오버스펙 — 실시간까지는 과함 |

특히 **Realtime으로 참여자 목록을 실시간 갱신**하는 기능은 멋져 보이지만, 지금 MVP에는 과한 사양이라 제거합니다. 무엇을 만들지만큼이나 **무엇을 안 만들지 정하는 것**도 중요한 설계 판단입니다.
기능을 덜어내면 그만큼 빠르고 단순하게 핵심에 집중할 수 있습니다.

### 계획을 두 번 거른다: 태스크 매니저 + Plan 모드

이번 챕터에서 가장 강조되는 노하우입니다. 강사는 Shrimp Task Manager로 이미 계획을 세웠더라도, **구현 직전에 Claude Code의 Plan 모드로 한 번 더 상세 계획을 살펴봅니다.**

> "Taskmaster AI나 Shrimp Task Manager를 통해서 계획을 세웠다고 하더라도 구현을 하기 전에 플랜 모드를 활용해서 상세 계획을 한 번 살펴보는 편이에요."

> "그러면 바로 바이패스 퍼미션 모드로 구현할 때보다 제가 원하는 대로 구현할 수 있는 확률이 올라가죠."

즉, 계획을 **이중으로 거르는** 것입니다.

1. **Shrimp Task Manager** — 큰 흐름의 작업 계획·분할
2. **Claude Code Plan 모드** — 구현 직전, 세부 계획 재점검
3. **Bypass permissions** — 검증된 계획대로 실행

한 번 더 거르는 이 과정이 번거로워 보여도, **원하는 대로 구현될 확률**을 눈에 띄게 높여 줍니다. (참고로 강사는 Claude Code가 Sonnet 4.5로 업그레이드되며 Plan 모드의 퀄리티가 한층
좋아졌다고도 언급합니다.)

### 참여 플로우 테스트: 시크릿 탭으로 다른 계정 흉내내기

기능을 구현한 뒤에는 실제 **초대 → 참여 흐름**을 테스트합니다. 방법이 현실적입니다.

- 주최자 계정으로 이벤트를 생성하고 **공유 URL을 복사**한 뒤,
- **시크릿 탭**에서 그 링크로 접속해 **다른 계정으로 로그인**하여 참여자 입장을 재현합니다.

참여하기 버튼을 누르면 참여자 수가 올라가는 것을 확인할 수 있습니다. 시크릿 탭은 별도 계정 환경을 손쉽게 흉내 내는 좋은 테스트 도구입니다.

### 핵심 개선: 로그인 후 '원래 가려던 페이지'로 복귀

테스트에서 가장 중요한 문제가 드러납니다. **초대 링크를 클릭했는데 로그인이 안 되어 있으면**, 로그인 후 엉뚱하게 **홈으로** 가버리는 것입니다. 초대받은 사람 입장에선 황당한 경험이죠. 원래 가려던 **초대
페이지로 돌아와야** 합니다.

해결의 열쇠는 **쿼리 파라미터(query parameter)** 입니다.

> **동작 원리.** 로그인 페이지로 보낼 때 URL에 `?redirect=/초대페이지경로` 같은 **복귀 주소를 쿼리 파라미터로 실어** 보냅니다. 로그인이 끝나면 그 값을 읽어 원래 페이지로 되돌립니다.

강사는 이 개선을 **단계별 사고(CoT) 방식**으로 계획하고, **테스트 계정(주최자·참여자)** 을 프롬프트에 명시해 Playwright MCP로 검증합니다. 다룬 테스트 시나리오는 다음과 같습니다.

| 시나리오                    | 기대 동작                |
|-------------------------|----------------------|
| 초대 링크 접속 → 일반 로그인       | 로그인 후 **초대 페이지로 복귀** |
| 초대 링크 접속 → 구글 OAuth 로그인 | 로그인 후 **초대 페이지로 복귀** |
| 초대 없이 직접 로그인            | 평소대로 **홈으로 이동**      |

처음엔 로그인 페이지로 보낼 때 **쿼리 파라미터가 빠지는** 문제가 있었지만, `redirect` 파라미터를 추가하자 로그인 후 초대 페이지로 정확히 복귀하는 것을 확인합니다.

### 미루는 것도 결정이다: 참여자 목록 보류

참여 기능은 완성됐지만, **이벤트 상세 페이지의 참여자 목록은 아직 표시되지 않습니다.** 강사는 이를 무리해서 지금 끝내지 않고 **의도적으로 보류**합니다. 참여자 프로필을 제대로 보여주려면 처리할 게 많아서,
**나중에 회원가입 시 닉네임 설정 등과 함께 일괄 처리**하는 편이 효율적이기 때문입니다.

이는 앞서 배운 `ISSUES` 관리와 같은 맥락입니다. **지금 할 일과 나중에 묶어서 할 일을 구분**하는 것도 실력입니다.

### 정리하며

이번 챕터에서는 이벤트 참여 시스템을 만들며 다음을 경험했습니다.

1. **Realtime·상태 자동 관리** 등 오버스펙을 걷어내 핵심에 집중하고,
2. **Shrimp + Plan 모드**로 계획을 이중으로 걸러 구현 정확도를 높였으며,
3. **쿼리 파라미터**로 로그인 후 초대 페이지 복귀 플로우를 완성하고,
4. 참여자 목록은 **나중에 일괄 처리**하기로 합리적으로 보류했습니다.

가장 기억할 메시지는 **'판단'** 입니다. 무엇을 만들고(참여 플로우), 무엇을 안 만들고(Realtime), 무엇을 미룰지(참여자 목록)를 결정하는 것 — AI가 코드를 짜 주는 시대일수록, 이 **방향을
정하는 사람의 판단**이 결과를 좌우합니다.

## 관리자 구현 및 Playwright MCP 테스트

사용자용 기능을 거의 마쳤으니, 이번에는 **관리자 대시보드**를 만들 차례입니다. 이벤트·사용자 통계를 한눈에 보는 관리 화면을 구현하고, **Playwright MCP로 E2E 테스트**까지 돌려 봅니다. 이
과정에서 데이터베이스 보안과 관련된 흥미로운 함정 하나를 만나게 됩니다.

### 첫 관문: 관리자 권한 부여

관리자 페이지에 들어가려니 **"권한이 없다"** 는 메시지가 막아섭니다. 당연합니다. 아직 관리자로 지정된 계정이 없으니까요. 권한을 주려면 Supabase 대시보드에서 `profiles` 테이블의 특정 사용자 *
*`role`을 `admin`으로 변경**하면 됩니다.

> "관리자 권한을 주기 위해서는 Supabase 대시보드로 와서요."

그런데 여기서 예상치 못한 일이 벌어집니다.

### 함정: role 변경을 막는 트리거 함수

`role`을 `admin`으로 직접 수정하려 하자 **오류가 발생**합니다. 강사는 원인을 Claude Code에게 물어 분석하고, 범인을 찾아냅니다. 바로 **`role` 변경을 제한하는 Supabase 트리거
함수**입니다.

생각해 보면 이건 **버그가 아니라 보안 기능**입니다. 누군가 함부로 자기 권한을 `admin`으로 바꾸지 못하도록 막아 두는 안전장치죠. 그래서 해결책도 이 보안을 존중하는 방식입니다.

> **해결 흐름** (SQL Editor에서)
> 1. `role` 변경을 막는 **트리거를 임시 비활성화**(disable)
> 2. 해당 사용자의 `role`을 **`admin`으로 수정**
> 3. 트리거를 **다시 활성화**(enable)

보안 장치를 잠깐 열고, 작업하고, 다시 닫는 — 무작정 트리거를 지워 버리는 게 아니라 **잠시 껐다 켜는** 신중한 접근입니다.

### 더 깊은 계획을 위한 도구: ultrathink와 flat mode

관리자 대시보드는 통계·쿼리·UI 연동이 얽힌 복잡한 작업입니다. 그래서 강사는 계획 단계에서 **두 가지 도구**를 동원합니다.

| 도구             | 역할                     |
|----------------|------------------------|
| **ultrathink** | 사고 예산을 늘려 더 깊고 꼼꼼하게 계획 |
| **flat mode**  | 더 구체적이고 평탄한 단계별 계획 요청  |

특히 **데이터베이스 마이그레이션**처럼 오류 위험이 큰 작업은 `flat mode`로 **한층 구체적인 계획**을 받아 대비합니다. 복잡하고 위험한 작업일수록, 계획에 더 많은 사고력을 투자하는 것입니다.

### 단계별 인프라 구축

계획이 서면 `bypass permission` 모드로 구현에 들어가는데, 한 번에 다 만들지 않고 **단계를 나눠** 쌓아 올립니다.

1. **인프라 구축** — DB 인덱스 생성, **RLS(Row Level Security) 정책** 생성, 관리자용 유틸리티 정의
2. **데이터 함수 구현** — 대시보드 통계, 이벤트 통계, 이벤트 목록, 유저 목록 함수
3. **대시보드 UI 연동** — `shrimp task manager`로 관리되는 3단계를 명시적으로 지정해 구현

이때도 **테스트에 쓸 계정을 미리 추가**해 두고 진행합니다. RLS 정책으로 접근 보안을 다지고, 통계 함수를 만들고, UI에 연결하는 — 백엔드부터 프론트까지 차곡차곡 이어지는 흐름입니다.

### Playwright MCP로 E2E 테스트

구현이 끝나면 **Playwright MCP로 E2E(End-to-End) 테스트**를 돌립니다. E2E 테스트는 실제 사용자처럼 **로그인부터 대시보드 진입까지 전 과정을 자동으로** 검증하는 방식입니다.

흥미로운 점은, 테스트 중 오류가 나면 **AI가 스스로 원인을 파악하고 해결**한다는 것입니다. 최종적으로 로그인이 성공하고 구현이 완료되는 것을 확인합니다.

> "다시 해당 계정으로 로그인을 하면 로그인이 되어 대시보드로 온 걸 확인할 수 있구요."

### 대시보드 기능 검증

마지막으로 관리자 계정으로 직접 로그인해 대시보드가 제대로 도는지 확인합니다.

> "해당 대시보드가 우리가 입력한 이벤트와, 최근 가입된 사용자, 그리고 통계 정보까지, 모든 정보가 잘 나오는 걸 확인할 수 있어요."

이벤트 관리, 사용자 관리, 통계 표시, SQL Editor를 통한 관리자 정보 확인까지 **기능별로 하나씩 검증**합니다. AI가 만든 화면을 그대로 믿지 않고, 항목마다 눈으로 확인하는 자세가 여기서도
이어집니다.

### 정리하며

이번 챕터에서는 관리자 대시보드를 만들며 다음을 경험했습니다.

1. **트리거를 껐다 켜며** 안전하게 관리자 권한을 부여하고,
2. **ultrathink·flat mode**로 복잡한 작업의 계획을 더 깊게 세웠으며,
3. **인프라 → 데이터 함수 → UI**의 단계로 대시보드를 쌓아 올리고,
4. **Playwright MCP E2E 테스트**로 전 과정을 자동 검증했습니다.

가장 기억할 장면은 **role 변경 오류**입니다. 그것이 버그가 아니라 **보안 트리거**임을 파악하고, 지우는 대신 **잠시 껐다 켜는** 방식으로 푼 것 — DB를 다룰 때 필요한 신중함을 잘 보여 줍니다.
다음 챕터에서는 Playwright MCP로 사용자 플로우 전체를 테스트해 봅니다.

## Playwright MCP로 사용자 플로우 테스트

기능을 하나씩 다 만들었으니, 이제 **전체가 매끄럽게 이어지는지** 확인할 차례입니다. 이번 챕터는 **Playwright MCP로 사용자 플로우 전체를 테스트**하는 방법을 다룹니다. 그리고 그 과정에서 **"
E2E 테스트 코드를 꼭 짜야 할까?"** 라는, 1인 개발자에게 특히 중요한 질문에 답합니다.

### E2E 테스트란 무엇인가

먼저 개념부터. **E2E(End-to-End) 테스트**는 사용자가 서비스를 처음 열어 원하는 작업을 끝낼 때까지의 **전 과정을 통째로** 검증하는 것입니다.

> "쉽게 말해서 사용자가 우리 서비스를 처음 열어서 원하는 작업을 완료하기까지의 전체 과정을 테스트하는 걸 의미합니다."

강사는 **쿠팡에서 물건을 사는 과정**을 예로 듭니다. 접속 → 검색 → 장바구니 → 결제까지 이어지는 흐름 전체가 끊김 없이 동작하는지 보는 것 — 그게 E2E입니다.

### E2E '코드'의 딜레마

전통적으로 E2E 테스트는 **사용자의 행동을 코드(스크립트)로 작성**합니다. 그런데 여기에 함정이 있습니다. **UI나 사용자 플로우가 바뀔 때마다 그 코드도 따라 고쳐야** 한다는 점입니다.

> "Claude Code를 활용해서 개발 생산성이 정말 뛰어난 지금과 같은 시대에 유지 보수를 하기가 정말 어렵죠."

특히 **1인 개발 환경**에서는 UI 개선과 비즈니스 로직 변경이 잦습니다. 그때마다 E2E 코드를 손보는 건 여간 번거로운 일이 아닙니다. AI 덕분에 개발 속도는 빨라졌는데, **테스트 코드 유지보수가 발목을
잡는** 역설이 생기는 것이죠.

### 대안: 코드 없이 브라우저를 직접 자동화

그래서 강사가 제안하는 방식이 **Claude Code + Playwright MCP로 브라우저를 직접 자동화**하는 것입니다. 테스트 코드를 작성하는 대신, **AI가 실제 브라우저를 조작하며 사용자처럼 테스트**
하게 만듭니다.

사용 흐름은 이렇습니다.

1. **Plan 모드**로 설정하고, "Playwright MCP를 활용해서 직접 브라우저로 테스트를 진행해주세요"라고 요청
2. **사용자 워크플로우와 테스트 계정**을 함께 입력
3. 개발 서버(`npm run dev`)를 자동 실행하고 브라우저를 띄움
4. 테스트 개요를 확인한 뒤 **bypass permissions** 모드로 실행

UI가 바뀌어도 **테스트 코드를 고칠 필요가 없다**는 게 핵심입니다. AI가 그때그때 화면을 보고 판단하기 때문입니다.

### 실전 테스트에서 잡아낸 크리티컬 버그

시연에서 AI는 로그인부터 시작해 전 과정을 자동으로 밟아 갑니다.

> **테스트한 사용자 플로우**
> 로그인 → 새 이벤트 생성 → 상세 페이지 진입 → 초대 링크 복사 → 초대 링크 접속

그런데 이벤트 생성 단계에서 **오류가 발생**합니다. AI는 원인이 **RLS(Row Level Security) 정책** 문제임을 파악하고, 정책을 수정한 뒤 다시 검증해 이벤트를 성공적으로 등록합니다. 단순한
기능 확인을 넘어, **크리티컬한 보안 버그를 테스트 도중 발견하고 즉시 고친** 것입니다.

이것이 자동화 테스트의 진짜 가치입니다. 손으로 일일이 클릭했다면 놓쳤을 수도 있는 문제를, **전체 플로우를 끝까지 밟아 보며** 잡아낸 셈입니다.

### 그래서, E2E 코드는 영영 필요 없을까?

여기서 핵심 질문이 나옵니다.

> "그러면 E2E 코드를 작성하지 말고 그냥 브라우저 자동화로 테스트를 진행하면 될까요?"

강사의 답은 **"상황에 따라 다르다"** 입니다. 둘은 대체재가 아니라 **역할이 다른 도구**입니다.

| 상황                           | 권장 방식                                   |
|------------------------------|-----------------------------------------|
| **결제·핵심 비즈니스 로직** 등 크리티컬한 부분 | **E2E 테스트 코드** 작성 (안정적·반복 검증)           |
| **UI·워크플로우를 자주 개선**하는 부분     | **Playwright MCP 브라우저 자동화** (유지보수 부담 ↓) |

즉, **절대 깨지면 안 되는 곳**은 견고한 테스트 코드로 지키고, **자주 바뀌는 곳**은 AI 자동화로 유연하게 검증하라는 균형 잡힌 전략입니다.

### 반복 테스트는 커스텀 커맨드로

같은 테스트를 주기적으로 돌려야 한다면, **커스텀 커맨드(custom command)** 로 테스트 프롬프트를 저장해 두고 반복 실행할 수 있습니다. 매번 긴 요청을 다시 입력할 필요 없이, 명령 하나로 전체
플로우 테스트를 굴리는 것입니다.

### 정리하며

이번 챕터에서는 완성된 앱을 **사용자 플로우 단위로 검증**하는 법을 배웠습니다.

1. **E2E 테스트**의 개념(전 과정 통합 검증)을 이해하고,
2. **E2E 코드의 유지보수 부담**이라는 현실적 한계를 짚은 뒤,
3. **Playwright MCP 브라우저 자동화**로 코드 없이 테스트하고,
4. 그 과정에서 **RLS 크리티컬 버그**까지 발견·수정했으며,
5. **크리티컬은 코드 / 가변적인 곳은 자동화**라는 테스트 전략을 정리했습니다.

핵심 메시지는 **"도구를 상황에 맞게 쓰라"** 입니다. AI 브라우저 자동화는 강력하지만 만능은 아닙니다. 무엇을 견고하게 지키고 무엇을 유연하게 검증할지 — 그 균형을 아는 것이 좋은 테스트 전략의 시작입니다.
이로써 이벤트 관리 앱의 구현과 테스트가 모두 마무리되었습니다.
